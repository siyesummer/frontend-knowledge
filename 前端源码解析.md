## Vue 3 响应式系统

### 一、惰性深度响应：getter 中为何还要 `reactive(res)`

#### 源码片段（`packages/reactivity/src/baseHandlers.ts`）

```typescript
function createGetter(isReadonly = false, shallow = false) {
  return function get(target: Target, key: string | symbol, receiver: object) {
    // ... 省略 ReactiveFlags、数组特殊处理 ...

    const res = Reflect.get(target, key, receiver)

    if (!isReadonly) {
      track(target, TrackOpTypes.GET, key)
    }

    if (shallow) {
      return res                       // 浅响应：直接返回，不再递归
    }

    if (isRef(res)) {
      return targetIsArray && isIntegerKey(key) ? res : res.value
    }

    if (isObject(res)) {
      // ★ 关键：访问到嵌套对象时，再把它也包成 Proxy
      return isReadonly ? readonly(res) : reactive(res)
    }

    return res
  }
}
```

#### 核心问题

> **为什么 `res` 是对象时还要再调一次 `reactive(res)`？**

#### 1. Proxy 的天生限制：只能拦截一层

```javascript
const raw = { user: { name: 'Tom' } };
const proxy = new Proxy(raw, {
  get(target, key) {
    console.log('get:', key);
    return target[key];                // 直接返回原始对象 user
  }
});

proxy.user.name = 'Jerry';
// 输出：get: user
// ❌ 没有触发 set！因为 user 是原始对象，跟 Proxy 没关系
```

`proxy.user` 拿到的是**原始的 `{ name: 'Tom' }`**，对它的属性赋值与外层 Proxy 完全无关，set 拦截器根本不会触发。

#### 2. Vue 3 的解决方案：访问时再包一层

走完 `isObject(res) → reactive(res)` 之后：

```javascript
const state = reactive({ user: { name: 'Tom' } });

state.user
//   ① 触发外层 Proxy 的 get('user')
//   ② res = { name: 'Tom' } 是普通对象
//   ③ reactive(res) → 返回一个新的 Proxy

state.user.name = 'Jerry';
//   ① state.user      → 拿到 user 的 Proxy
//   ② .name = 'Jerry' → 命中内层 Proxy 的 set，触发依赖更新 ✅
```

#### 3. 为什么是"惰性"，而不是初始化时递归？

| 时机 | Vue 2 (`Object.defineProperty`) | Vue 3 (`Proxy`) |
|------|------|------|
| 创建响应式对象时 | **递归遍历所有嵌套属性**，逐个 `defineProperty` | 只代理最外层 |
| 嵌套对象首次访问 | 已是响应式（启动时就处理过） | **此时才调用 `reactive()`** |
| 新增属性 | 检测不到，需 `Vue.set` | Proxy 自动覆盖新增 key |
| 大对象初始化开销 | 高（递归整个对象树） | 低（按需转换） |

**惰性的四大好处**：

1. **性能**：永远不访问的深层属性，永远不会被转换，节省内存与初始化时间
2. **支持动态新增属性**：新加的对象 key 也能被自动代理
3. **避免循环引用死循环**：递归方式遇到 `a.b.a === a` 会爆栈，惰性天然规避
4. **支持大数据结构懒加载**：例如几万节点的树，只渲染可见部分

#### 4. `reactiveMap` 缓存：同一对象只转换一次

担心每次 `state.user` 都生成新 Proxy？源码用 **WeakMap** 缓存：

```typescript
// reactive.ts 源码（简化）
function reactive(target) {
  const existingProxy = reactiveMap.get(target);
  if (existingProxy) return existingProxy;  // ← 命中缓存就直接返回

  const proxy = new Proxy(target, mutableHandlers);
  reactiveMap.set(target, proxy);
  return proxy;
}
```

验证：

```javascript
state.user === state.user;   // true，每次拿到的是同一个 Proxy
```

WeakMap 还能让 GC 在原始对象被回收时自动清理对应 Proxy，避免内存泄漏。

#### 5. `shallow` 直接 return 的对照证明

```typescript
if (shallow) {
  return res;     // 浅响应：嵌套对象不再转换
}
```

`shallowReactive` 故意只代理一层，深层修改不触发响应——反向说明：**不调 `reactive(res)` 就没有深层响应**。

#### 6. 完整的访问链路

先约定原始对象的命名，便于在流程中区分两个不同的 `target`：

```javascript
const rawRoot = { user: { name: 'Tom' } };   // 根对象（原始）
const rawUser = rawRoot.user;                // 嵌套对象（原始） = { name: 'Tom' }

const state = reactive(rawRoot);             // 外层 Proxy 的 target = rawRoot
//                                              内层 Proxy 的 target = rawUser（在首次访问时生成）
```

```
   外层 Proxy ─── target: rawRoot  = { user: rawUser }
                                      │
                                      ▼
   内层 Proxy ─── target: rawUser  = { name: 'Tom' }

state.user.name = 'Jerry'
   │
   ├─ ① 触发【外层 Proxy】的 get('user')
   │       ├─ 此处 target = rawRoot（外层 Proxy 包装的原始对象）
   │       ├─ res = Reflect.get(rawRoot, 'user') = rawUser  // { name: 'Tom' }
   │       ├─ track(rawRoot, 'user')                        // 把当前 effect 收集到
   │       │                                                //   depsMap(rawRoot) → 'user' 这一项
   │       ├─ isObject(res) ✅
   │       └─ 返回 reactive(rawUser) → 内层 Proxy
   │           （reactiveMap 中以 rawUser 为 key 缓存，命中则复用）
   │
   └─ ② 在【内层 Proxy】上 set('name', 'Jerry')
           ├─ 此处 target = rawUser（内层 Proxy 包装的原始对象）
           ├─ Reflect.set(rawUser, 'name', 'Jerry')  // 真正改的是 rawUser.name
           ├─ trigger(rawUser, 'name')               // 派发依赖：
           │                                         //   depsMap(rawUser) → 'name' 项的 effects
           └─ 组件重新渲染 ✅
```

#### 关键点：两个 target 是不同的原始对象

| 阶段 | Proxy 层级 | `target` 是谁 | `key` | 维护哪张依赖表 |
|------|-----------|---------------|-------|----------------|
| ① `state.user` | 外层 Proxy | `rawRoot` （根对象本身） | `'user'` | `targetMap.get(rawRoot)` |
| ② `.name = 'Jerry'` | 内层 Proxy | `rawUser` （嵌套对象本身） | `'name'` | `targetMap.get(rawUser)` |

> **`target` 始终是该 Proxy 包装的"原始对象"，而不是 Proxy 自身**。
> Vue 在全局用一张 `WeakMap<原始对象, Map<key, Set<effect>>>`（即 `targetMap`）来维护依赖，
> 所以两层访问对应**两条独立的依赖记录**：
> - `targetMap → rawRoot → 'user' → [effects]`
> - `targetMap → rawUser → 'name' → [effects]`
>
> 这也是为什么改 `state.user = {…}`（整体替换）和改 `state.user.name`（修改内部）
> 触发的是**不同 key 上的依赖**，互不干扰。

#### 7. 一句话总结

> Proxy 只能拦截一层属性，所以 Vue 3 在 getter 中**惰性地**把访问到的对象属性也包装成 Proxy。
> 这样既实现了深度响应，又避免了 Vue 2 那种初始化时递归整个对象树的性能开销，同时天然支持动态新增的嵌套属性。

### 二、ReactiveEffect 类与 TS 参数属性

#### 源码片段（`packages/reactivity/src/effect.ts`）

```typescript
export class ReactiveEffect<T = any> {
  active = true
  deps: Dep[] = []
  parent: ReactiveEffect | undefined = undefined

  constructor(
    public fn: () => T,                              // ← 参数属性
    public scheduler: EffectScheduler | null = null, // ← 参数属性
    scope?: EffectScope                              // ← 普通参数，不挂 this
  ) {
    recordEffectScope(this, scope)
  }

  run() {
    if (!this.active) return this.fn()
    // ... 维护 activeEffect / parent / trackOpBit ...
    try {
      this.parent = activeEffect
      activeEffect = this
      shouldTrack = true
      return this.fn()                // ✅ 直接调用，因为 this.fn 已被自动赋值
    } finally {
      // 还原状态
    }
  }
}
```

#### 核心问题

> **`run()` 里为什么可以直接 `this.fn()`？类里并没有显式声明 `fn` 字段啊。**

#### 答案：TS 的"参数属性"语法糖

`public fn: () => T` 写在构造函数参数上，TS 会自动展开为"声明 + 构造期赋值"两步：

```typescript
// 你写的（参数属性）
constructor(public fn: () => T) {}

// 等价的展开写法
class ReactiveEffect<T> {
  fn: () => T;
  constructor(fn: () => T) {
    this.fn = fn;     // ← 编译器替你生成
  }
}
```

编译后的 JS：

```javascript
class ReactiveEffect {
  constructor(fn, scheduler = null, scope) {
    this.active = true;
    this.deps = [];
    this.parent = undefined;
    this.fn = fn;                 // ← 由参数属性生成
    this.scheduler = scheduler;   // ← 由参数属性生成
    recordEffectScope(this, scope);
    // scope 没有 this.scope，因为它不是参数属性
  }
}
```

#### 修饰符对比

| 修饰符 | 自动建属性 | 外部可访问 | 子类可访问 | 可重新赋值 |
|--------|-----------|-----------|-----------|-----------|
| `public` | ✅ | ✅ | ✅ | ✅ |
| `protected` | ✅ | ❌ | ✅ | ✅ |
| `private` | ✅ | ❌ | ❌ | ✅ |
| `readonly` | ✅ | ✅（默认 public） | ✅ | ❌（仅构造期可赋值） |

#### 三种参数形态

```typescript
class Demo {
  constructor(
    public a: number,    // ① 参数属性 → this.a 可用
    b: number,           // ② 普通参数 → 仅构造期可见，外部访问不到
    private c: number    // ③ 私有参数属性 → this.c 仅类内部访问
  ) {
    console.log(b);      // ✅ 构造期能用
  }
}
```

#### 一句话总结

> `public fn: () => T` 是 TS 的**参数属性**语法糖，编译时会自动生成 `this.fn = fn` 的赋值，所以 `run()` 能直接 `this.fn()`。Vue 源码大量使用此语法以保持代码紧凑。

### 三、`isRef` 是怎么判断的：类型谓词 + 标志位

#### 源码片段（`packages/reactivity/src/ref.ts`）

```typescript
export function isRef<T>(r: Ref<T> | unknown): r is Ref<T>
export function isRef(r: any): r is Ref {
  return !!(r && r.__v_isRef === true)
}
```

#### 1. 函数签名拆解

```typescript
export function isRef<T>(r: Ref<T> | unknown): r is Ref<T>
//                       └─── 入参类型 ───┘    └─ 类型谓词 ─┘
```

| 部分 | 作用 |
|------|------|
| `<T>` | 泛型，承载 Ref 内部包裹的真实值类型 |
| `r: Ref<T> \| unknown` | 入参可以是任何东西（`unknown`），也可能就是 `Ref<T>` |
| `r is Ref<T>` | **类型谓词返回值** —— 告诉 TS："如果此函数返回 `true`，那么 `r` 就是 `Ref<T>`" |

#### 2. 类型谓词（Type Predicate）的作用

普通函数返回 `boolean` 时，TS **不会**自动收窄类型；类型谓词函数返回 `true` 后，TS 会在 `if` 分支内**收窄变量类型**。

```typescript
// ❌ 普通 boolean 返回值：TS 不收窄
function isRefBad(r: unknown): boolean {
  return r != null && (r as any).__v_isRef === true;
}
const x: unknown = ref(1);
if (isRefBad(x)) {
  x.value;   // ❌ Error: x 仍是 unknown
}

// ✅ 类型谓词：TS 自动收窄
function isRef(r: unknown): r is Ref {
  return r != null && (r as any).__v_isRef === true;
}
const y: unknown = ref(1);
if (isRef(y)) {
  y.value;   // ✅ OK: y 被收窄为 Ref<unknown>
}
```

这就是 Vue 源码里大量这样写的原因——`isRef` 之后才能安全地 `.value`。

#### 3. 运行时实际怎么判断

类型谓词只是**编译期承诺**，运行时还得真识别一个对象是不是 Ref。Vue 3 的实现非常简短：

```typescript
export function isRef(r: any): r is Ref {
  return !!(r && r.__v_isRef === true)
}
```

两件事：

1. `!!r` —— 排除 `null` / `undefined` / `0` / `''` 等 falsy 值
2. `r.__v_isRef === true` —— 检查对象上的标志位

#### 4. `__v_isRef` 这个标志位是从哪来的？

Vue 在 `RefImpl` 类里写死了这个属性：

```typescript
class RefImpl<T> {
  private _value: T
  private _rawValue: T

  public dep?: Dep
  public readonly __v_isRef = true   // ★ 关键标记位

  constructor(value: T, public readonly __v_isShallow: boolean) {
    this._rawValue = __v_isShallow ? value : toRaw(value)
    this._value   = __v_isShallow ? value : toReactive(value)
  }

  get value() { /* track + 返回 _value */ }
  set value(newVal) { /* trigger + 更新 _value */ }
}

export function ref<T>(value: T): Ref<UnwrapRef<T>> {
  return createRef(value, false)
}
```

所以**所有由 `ref()` / `shallowRef()` / `customRef()` 创建出的对象，都带有 `__v_isRef: true`**。`isRef` 直接读这个字段就够了，不需要 `instanceof RefImpl`。

#### 5. 为什么用"标志位"而不是 `instanceof`？

| 方案 | 缺点 |
|------|------|
| `instanceof RefImpl` | 必须导出 `RefImpl` 类；多包/多副本环境（monorepo、microfrontend）下 `instanceof` 会失效 |
| `__v_isRef === true` | 只依赖一个约定的字符串字段，跨包/跨实例都能识别；用户的 `customRef` 也能遵守约定 |

Vue 整套约定标志位：

| 标志 | 含义 |
|------|------|
| `__v_isRef` | 是 Ref |
| `__v_isReactive` | 是 reactive Proxy |
| `__v_isReadonly` | 是 readonly Proxy |
| `__v_isShallow` | 是浅响应 |
| `__v_raw` | 原始对象引用 |

`__v_` 前缀表示 Vue 内部约定，**用户代码不要碰**。

#### 6. 回到 `createGetter` 中的应用

```typescript
if (isRef(res)) {
  return targetIsArray && isIntegerKey(key) ? res : res.value
}
```

逻辑：

1. `isRef(res)` 运行时检查 `res.__v_isRef === true`
2. TS 在编译期把 `res` 收窄为 `Ref` 类型
3. 安全地 `res.value` —— 实现"自动解包 ref"的效果（template 中 `state.count` 等价于 `state.count.value`）
4. 数组 + 整数索引的例外，避免 `arr[0]` 被错误解包

#### 7. 一句话总结

> `r is Ref<T>` 是 TS 的**类型谓词**，让编译器在 `if` 分支内收窄类型；
> 运行时则通过检查对象上的 **`__v_isRef === true`** 标志位识别。
> 这种"约定字段 + 类型谓词"的组合，是 Vue 3 内部识别响应式对象的统一模式。

### 四、`computed` 的实现：lazy + dirty 双缓存机制

#### 0. 基础用法回顾

```typescript
import { ref, computed } from 'vue'

const count = ref(1)

// 1) 只读 computed（getter）
const double = computed(() => count.value * 2)
console.log(double.value)   // 2

// 2) 可写 computed（getter + setter）
const writable = computed({
  get: () => count.value * 2,
  set: (v) => { count.value = v / 2 }
})
writable.value = 10
console.log(count.value)    // 5
```

三大特性：
- **惰性求值（lazy）**：getter 不会立刻执行，只有 `.value` 被读时才运行
- **结果缓存（cache）**：依赖未变时，多次访问 `.value` 只返回缓存值，不重复计算
- **依赖变更自动失效（dirty）**：依赖的响应式数据变化后，下次访问会重算

---

#### 1. 入口函数 `computed()`：两种重载、四种参数

```typescript
// packages/reactivity/src/computed.ts
export function computed<T>(
  getterOrOptions: ComputedGetter<T> | WritableComputedOptions<T>,
  debugOptions?: DebuggerOptions,
  isSSR = false
) {
  let getter: ComputedGetter<T>
  let setter: ComputedSetter<T>

  const onlyGetter = isFunction(getterOrOptions)
  if (onlyGetter) {
    // 重载 ①：只读 computed —— 传入函数
    getter = getterOrOptions
    setter = __DEV__
      ? () => { console.warn('Write operation failed: computed value is readonly') }
      : NOOP                            // 生产环境直接无操作
  } else {
    // 重载 ②：可写 computed —— 传入 { get, set }
    getter = getterOrOptions.get
    setter = getterOrOptions.set
  }

  const cRef = new ComputedRefImpl(getter, setter, onlyGetter || !setter, isSSR)

  if (__DEV__ && debugOptions && !isSSR) {
    cRef.effect.onTrack   = debugOptions.onTrack
    cRef.effect.onTrigger = debugOptions.onTrigger
  }

  return cRef as any
}
```

参数解析：

| 参数 | 类型 | 说明 |
|------|------|------|
| `getterOrOptions` | 函数 或 `{ get, set }` | 决定只读还是可写 |
| `debugOptions` | `{ onTrack, onTrigger }` | 调试钩子，仅 dev 生效 |
| `isSSR` | boolean | 服务端渲染时禁用缓存（每次都重算，避免脏数据跨请求） |

`onlyGetter || !setter` 这个布尔值决定 `IS_READONLY` 标志：只传函数、或者 options 没传 setter，都视为只读。

---

#### 2. 核心类 `ComputedRefImpl` 完整字段表

```typescript
// packages/reactivity/src/computed.ts
class ComputedRefImpl<T> {
  public dep?: Dep = undefined                 // ① 下游订阅者集合
  private _value!: T                           // ② 缓存的计算结果
  public readonly effect: ReactiveEffect<T>    // ③ getter 被包成的 effect

  public readonly __v_isRef = true             // ④ 让 isRef() 识别
  public readonly [ReactiveFlags.IS_READONLY]: boolean  // ⑤ 是否只读

  public _dirty = true                         // ⑥ 是否需要重新计算
  public _cacheable: boolean                   // ⑦ 是否允许缓存（SSR 关）

  constructor(
    getter: ComputedGetter<T>,
    private readonly _setter: ComputedSetter<T>,
    isReadonly: boolean,
    isSSR: boolean
  ) {
    this.effect = new ReactiveEffect(getter, () => {
      if (!this._dirty) {
        this._dirty = true
        triggerRefValue(this)                  // 通知下游
      }
    })
    this.effect.computed = this                // ⑧ 反向引用：从 effect 找到 computed
    this.effect.active = this._cacheable = !isSSR
    this[ReactiveFlags.IS_READONLY] = isReadonly
  }

  get value() {
    const self = toRaw(this)                   // ⑨ 穿透 readonly 包装
    trackRefValue(self)                        // ⑩ 收集下游依赖
    if (self._dirty || !self._cacheable) {     // ⑪ dirty 或 SSR 时重算
      self._dirty = false
      self._value = self.effect.run()!
    }
    return self._value
  }

  set value(newValue: T) {
    this._setter(newValue)
  }
}
```

每个编号的细节：

| # | 字段/操作 | 用途 |
|---|-----------|------|
| ① `dep` | 下游 effect 集合 | 谁读了 `.value`，谁就被记进来；computed 变脏时挨个通知 |
| ② `_value` | 上一次的计算结果 | 不脏就直接返回它 |
| ③ `effect` | 包装 getter 的 ReactiveEffect | 用来在响应式系统中"代表 getter" |
| ④ `__v_isRef = true` | 让 `isRef()` / 模板解包识别它 | 模板里 `{{ c }}` 自动解包成 `c.value` |
| ⑤ `IS_READONLY` | 让 `isReadonly()` 识别只读 computed | `c.value = ...` 触发警告（dev） |
| ⑥ `_dirty` | 是否需要重算的标志 | 初始 `true`，首次读取时计算；scheduler 触发后变回 `true` |
| ⑦ `_cacheable` | SSR 中禁用缓存 | 服务端渲染每个请求独立，缓存会跨请求污染 |
| ⑧ `effect.computed = this` | 让 effect 反向认识它属于哪个 computed | 处理"嵌套 computed 死循环"等场景要用 |
| ⑨ `toRaw(this)` | 穿透外层 readonly Proxy | 只读 computed 在 setup 中可能被 `readonly()` 再包一层 |
| ⑩ `trackRefValue(self)` | 把当前 activeEffect 加入 `dep` | computed 作为"被订阅者"建立反向链 |
| ⑪ `_dirty \|\| !_cacheable` | 重算条件 | SSR 下 `_cacheable=false`，所以每次都重算 |

---

#### 3. 三大方向的数据流

```
                   ┌─────────────────────────────────────────┐
                   │            ComputedRefImpl              │
                   │                                         │
                   │  ┌────────────────────┐                 │
                   │  │  effect            │                 │
                   │  │  (ReactiveEffect)  │◄── 包装 getter   │
                   │  │                    │                 │
                   │  │  scheduler:        │                 │
                   │  │    _dirty = true   │                 │
                   │  │    triggerRefValue │                 │
                   │  └────────┬───────────┘                 │
                   │           │                             │
                   │           │ run() 时执行 getter         │
                   │           ▼                             │
                   │       getter 函数                       │
                   │           │                             │
                   │           │ 访问 count.value 等          │
                   │           ▼                             │
                   │       【上游依赖】                       │
                   │                                         │
                   │  _value: 缓存结果                       │
                   │  _dirty: 脏标记                         │
                   │  dep:    下游订阅者集合                  │
                   │           ▲                             │
                   │           │ trackRefValue                │
                   │  get value() ◄── 下游 effect 读取        │
                   └───────────┼─────────────────────────────┘
                               │
                               ▼
                       【下游 effect】
                       (render / watch / 另一个 computed)
```

**三条链路详解**：

##### 链路一：上游 → computed（**依赖收集**，computed 的 effect 作为订阅者）

> 「getter 内部读了谁，谁就把 computed.effect 记下来」

发生时机：调用 `effect.run()` 执行 getter 期间。

具体步骤：

```typescript
// 假设：
const count = ref(1)
const double = computed(() => count.value * 2)

// 当第一次访问 double.value 时：
get value() {
  trackRefValue(self)
  if (self._dirty) {
    self._dirty = false
    self._value = self.effect.run()      // ★ 这里
  }
  return self._value
}

// effect.run() 内部：
run() {
  // ① 把全局 activeEffect 指向自己（关键！）
  this.parent = activeEffect
  activeEffect = this          // activeEffect = double.effect
  shouldTrack = true

  // ② 执行 fn() 也就是 getter
  return this.fn()             // () => count.value * 2
  //          │
  //          └─ 读 count.value
  //                │
  //                └─ RefImpl 的 get value() 调用 trackRefValue(count)
  //                      │
  //                      └─ trackEffects(count.dep)
  //                            │
  //                            └─ count.dep.add(activeEffect)
  //                            │       即：count.dep.add(double.effect)  ✅
  //                            │
  //                            └─ double.effect.deps.push(count.dep)
  //                                  双向引用：effect 也记录自己订阅了哪些 dep
}
```

**双向数据结构**：

```
   count (RefImpl)                       double (ComputedRefImpl)
   ┌─────────────┐                       ┌─────────────────────┐
   │ _value: 1   │                       │ _value: undefined   │
   │ dep: Set ───┼──► [ double.effect ]  │ effect: ReactiveEffect
   └─────────────┘                       │   └ deps: [ count.dep ]
                                         └─────────────────────┘
                  ▲                                  │
                  └──────── 反向引用 ────────────────┘
```

- `count.dep` 持有所有"读过 count.value 的 effect"
- `double.effect.deps` 记录自己订阅了哪些 dep（用于 `stop()` 时反清理）

##### 链路二：computed → 下游（**被订阅**，computed 自己作为被订阅者）

> 「谁读了 computed.value，谁就被 computed.dep 记下来」

发生时机：任何外部 effect（render / watch / 另一个 computed）读 `double.value` 时。

具体步骤：

```typescript
// 假设组件渲染函数：
function render() {
  return h('div', double.value + ' 元')   // ★ 读 double.value
}

// 组件的 render effect 跑起来后：
new ReactiveEffect(render).run()
//   │
//   ├─ activeEffect = renderEffect      ← 现在 activeEffect 是 render
//   ├─ 执行 render()
//   │     │
//   │     └─ 读 double.value
//   │           │
//   │           └─ ComputedRefImpl.get value()
//   │                 │
//   │                 ├─ trackRefValue(self)   ★ 关键
//   │                 │     │
//   │                 │     └─ trackEffects(self.dep)
//   │                 │           │
//   │                 │           └─ self.dep.add(activeEffect)
//   │                 │                即：double.dep.add(renderEffect) ✅
//   │                 │
//   │                 ├─ if (_dirty) effect.run()
//   │                 └─ return _value
```

**注意 `dep` 是延迟创建的**（`dep?: Dep = undefined`）：第一次 `trackRefValue` 时才 `new Dep()`，没人订阅则不创建，节省内存。

**关键陷阱：嵌套读取时 activeEffect 是哪个？**

```typescript
// 链式 computed
const a = ref(1)
const b = computed(() => a.value + 1)
const c = computed(() => b.value + 1)
```

调用栈展开：

```
render effect.run()
  activeEffect = renderEffect
  │
  └─ 读 c.value
       │
       ├─ trackRefValue(c)  → c.dep.add(renderEffect) ✅
       │
       └─ c.effect.run()
            this.parent   = renderEffect         ← 保存上一层
            activeEffect  = c.effect             ← 切换
            │
            └─ 执行 () => b.value + 1
                  │
                  └─ 读 b.value
                       │
                       ├─ trackRefValue(b) → b.dep.add(c.effect) ✅
                       │                       （此刻 activeEffect 是 c.effect）
                       │
                       └─ b.effect.run()
                            this.parent  = c.effect
                            activeEffect = b.effect
                            │
                            └─ 执行 () => a.value + 1
                                  │
                                  └─ 读 a.value
                                       └─ a.dep.add(b.effect) ✅
                            │
                            finally: activeEffect = this.parent = c.effect  ← 还原
                       │
                  finally: activeEffect = this.parent = renderEffect  ← 还原
```

这就是 `ReactiveEffect.run()` 里 `parent` 链的作用——支持任意深度的嵌套读取，每层结束后能精准还原上一层的 `activeEffect`。

最终的依赖图：

```
a.dep      = { b.effect }
b.dep      = { c.effect }
c.dep      = { renderEffect }
```

##### 链路三：上游变化 → 标脏 → 通知下游（**触发更新**）

> 「上游 trigger 时，computed 的 scheduler 介入：不算，只标脏 + 通知下游」

发生时机：上游（如 `count.value = 5`）被赋值时。

具体步骤：

```typescript
// 触发源头：
count.value = 5

// RefImpl 的 set value()：
set value(newVal) {
  if (hasChanged(newVal, this._rawValue)) {
    this._rawValue = newVal
    this._value    = newVal
    triggerRefValue(this, newVal)        // ★
  }
}

// triggerRefValue → triggerEffects(count.dep)
function triggerEffects(dep: Dep) {
  for (const effect of dep) {
    // 关键分支：computed 的 effect 有 scheduler，普通 render effect 也有 scheduler
    if (effect.scheduler) {
      effect.scheduler()                  // ★ 不走 effect.run()
    } else {
      effect.run()
    }
  }
}
```

`count.dep` 里的 `double.effect.scheduler` 被调用：

```typescript
// computed 创建时设置的 scheduler
scheduler = () => {
  if (!this._dirty) {              // ① 守卫：已脏就别再走一遍
    this._dirty = true             // ② 标记 computed 失效
    triggerRefValue(this)          // ③ 通知 computed.dep 里的所有下游
  }
}

// triggerRefValue(double) → 遍历 double.dep
//   └─ renderEffect.scheduler() （或 .run()）
//         └─ 调度器队列中加入 renderEffect，下一个 tick 执行 render
//               └─ render() 中再次读 double.value
//                     └─ _dirty === true，重新执行 effect.run() 得到 10
```

**完整时序图**：

```
   count.value = 5
        │
        ├─ trigger(count)
        │     │
        │     └─ 遍历 count.dep
        │           └─ double.effect.scheduler()
        │                 │
        │                 ├─ _dirty = true        ← 标脏，不算
        │                 │
        │                 └─ triggerRefValue(double)
        │                       │
        │                       └─ 遍历 double.dep
        │                             └─ renderEffect.scheduler()
        │                                   │
        │                                   └─ queueJob(render)   ← 入队，下个 tick 执行
        │
        ▼ （下个微任务）
   render() 执行
        │
        └─ 读 double.value
              │
              ├─ trackRefValue(double)  幂等收集
              │
              └─ _dirty === true → effect.run() 重算 → _value = 10
                    │
                    └─ effect.run() 内 fn() 读 count.value
                          └─ count.dep.add(double.effect)  幂等
                          └─ 返回 5 * 2 = 10
```

**三处守卫保证不重复劳动**：

| 守卫 | 位置 | 目的 |
|------|------|------|
| `if (!this._dirty)` | computed scheduler | 上游连续变化时，只通知下游一次 |
| `if (self._dirty)` 才 `effect.run()` | `get value()` | 没人改过的话直接返回缓存 |
| `Set` 自带去重 | `count.dep` / `double.dep` | 同一个 effect 多次 track 也只存一次 |

---

##### 三链路对照速记

| 链路 | 触发时机 | 操作的数据结构 | 谁动了谁 |
|------|----------|---------------|----------|
| 上游 → computed | `effect.run()` 执行 getter 时 | `上游.dep` + `computed.effect.deps` | 上游记下 computed.effect |
| computed → 下游 | 外部读 `computed.value` 时 | `computed.dep` | computed 记下外部 effect |
| 上游变化 → 标脏 → 通知下游 | 上游被 set 时 | `_dirty` 标志位 + 两次 trigger | scheduler 串联两次通知 |

#### 4. 时序：从创建到首次读取再到依赖变化

##### 阶段 A：创建

```typescript
const count = ref(1)
const double = computed(() => count.value * 2)
```

```
new ComputedRefImpl(() => count.value * 2, NOOP, true, false)
  │
  ├─ effect = new ReactiveEffect(getter, scheduler)
  │     └─ ReactiveEffect 内部 fn = getter，但 ★不会立即运行★
  │
  ├─ effect.computed = this
  ├─ _dirty = true
  ├─ _value = undefined
  └─ dep   = undefined
```

##### 阶段 B：首次访问 `double.value`

```
double.value
  │
  ├─ get value() 触发
  │     ├─ self = toRaw(this) = double
  │     │
  │     ├─ trackRefValue(self)
  │     │     └─ 把当前 activeEffect（如 render effect）加入 self.dep
  │     │
  │     ├─ _dirty === true？  ✅
  │     │     ├─ self._dirty = false
  │     │     └─ self._value = effect.run()
  │     │           └─ effect.run() 内部:
  │     │                ├─ activeEffect = effect    ← 关键！
  │     │                ├─ 执行 fn() = () => count.value * 2
  │     │                │     └─ 读 count.value
  │     │                │         └─ count.dep.add(effect) ✅ 上游依赖建立
  │     │                └─ 返回 2
  │     │
  │     └─ return 2
  │
  └─ 表达式得到 2
```

此时的关系：
- `count.dep` 包含 `double.effect`
- `double.dep` 包含外部 render effect

##### 阶段 C：再次读取（依赖未变）

```
double.value
  │
  ├─ get value()
  │     ├─ trackRefValue(self)         （依然收集依赖，幂等）
  │     ├─ _dirty === false           ✅ 不重算
  │     └─ return self._value = 2
```

##### 阶段 D：依赖发生变化

```typescript
count.value = 5
```

```
ref setter
  │
  └─ trigger(count, 'value')
        │
        └─ 遍历 count.dep 中的 effects
              │
              └─ 找到 double.effect
                    │
                    └─ ★ 如果 effect 有 scheduler，调用 scheduler 而不是 run()
                          │
                          └─ scheduler():
                                 │
                                 ├─ if (!this._dirty)         ← 首次进入
                                 │     ├─ this._dirty = true
                                 │     └─ triggerRefValue(this)
                                 │           │
                                 │           └─ 遍历 double.dep
                                 │                 └─ 唤醒下游 render effect
```

> 注意：到这里 **getter 仍然没被执行**！只是改了脏标记。

##### 阶段 E：下游 effect 被唤醒后再读 `double.value`

```
render() 中读 double.value
  │
  └─ get value() → _dirty === true → 重新执行 effect.run() → 得到 10
```

#### 5. 关键设计点深入解析

##### ① 为什么 getter 要包成 `ReactiveEffect`？

不包成 effect 就**无法接入响应式系统**——getter 中读 `count.value` 时，依赖收集需要"当前活跃 effect"才能完成。`effect.run()` 内会临时把 `activeEffect = this`，再去执行 getter，让 getter 中的所有响应式读取都把这个 effect 当作订阅者收集起来。

##### ② 为什么 scheduler 只标 dirty 不直接重算？

> **真正的"惰性求值"**：依赖变了不代表结果一定被人用。立刻重算可能算了白算。

```typescript
const heavy = computed(() => doExpensiveCalc(count.value))
count.value++       // ① _dirty=true，但 doExpensiveCalc 没跑
count.value++       // ② _dirty 已经是 true，scheduler 早就 return 了
count.value++       // ③ 同上
console.log(heavy.value)   // ④ 此刻才真正算一次
```

注意 scheduler 里有个 `if (!this._dirty)` 的守卫：

```typescript
this.effect = new ReactiveEffect(getter, () => {
  if (!this._dirty) {        // ★ 已经脏了就不重复通知下游
    this._dirty = true
    triggerRefValue(this)
  }
})
```

这避免了：依赖连续变 100 次 → 通知下游 100 次 → 下游可能渲染 100 次的灾难。

##### ③ 双向身份：computed 既是 effect 又是 ref

| 身份 | 体现 | 数据结构 |
|------|------|----------|
| 作为 Ref（被订阅者） | `__v_isRef = true`，有 `.value`，有 `dep` | `dep: Set<ReactiveEffect>` |
| 作为 Effect（订阅者） | 持有 `effect: ReactiveEffect`，会被上游 `dep` 收集 | `effect.deps: Dep[]` |

这就是为什么 computed 能 "**串成链**"——一个 computed 既能订阅别的响应式数据，又能被别的 effect 订阅。

##### ④ 链式 computed 与"嵌套读时机"

```typescript
const a = ref(1)
const b = computed(() => a.value + 1)
const c = computed(() => b.value + 1)
```

**关键点**：`c.effect.run()` 执行 `() => b.value + 1` 时，读 `b.value` 触发 `trackRefValue(b)`，此时 `activeEffect === c.effect`，所以 **c.effect 被收进 b.dep**。

链路示意：

```
a.dep      → [b.effect]
b.dep      → [c.effect]
c.dep      → [render effect]
```

a 变化 → 触发 b.effect 的 scheduler → b._dirty=true 并 triggerRefValue(b) → 触发 c.effect 的 scheduler → c._dirty=true 并 triggerRefValue(c) → 唤醒 render → 读 c.value → 重算 c → 期间读 b.value 重算 b → 期间读 a.value 拿最新值。

中间值在没人读时不会重算，访问 `c.value` 时按需链式重算。

##### ⑤ `__v_isRef = true` 与模板自动解包

```vue
<template>
  <!-- 模板里 double 等价于 double.value -->
  <p>{{ double }}</p>
</template>
```

setup 返回的对象里，`double` 是 ComputedRefImpl。`isRef(double) === true`，所以模板编译器/Proxy setup state 在访问时自动 `.value`。

而 reactive 包装一个含 computed 的对象时，访问也会自动解包：

```typescript
const state = reactive({ double })
state.double   // 自动解包 → 数字，不是 ComputedRefImpl
```

##### ⑥ SSR 模式（`isSSR=true`）禁用缓存

```typescript
this.effect.active = this._cacheable = !isSSR
```

服务端渲染中每个请求是独立的，如果 computed 跨请求复用缓存值会发生数据污染。SSR 下：

- `_cacheable=false`：每次读 `.value` 都重算
- `effect.active=false`：`run()` 不会做完整的依赖追踪（避免在 SSR 上构建无用依赖图）

效果上等同于一个普通函数调用。

##### ⑦ `toRaw(this)` 为什么必要？

```typescript
get value() {
  const self = toRaw(this)
  // ...
}
```

考虑 `readonly(computed(...))` 这种嵌套：外层 readonly Proxy 会再代理一层。如果 `this` 是 Proxy，访问 `this._dirty` 等内部字段会走 Proxy 的 getter（性能差且可能拿不到真正字段）。用 `toRaw(this)` 拿到原始的 `ComputedRefImpl` 实例，直接操作字段。

##### ⑧ Setter 的写：不重算，只转写

```typescript
set value(newValue: T) {
  this._setter(newValue)
}
```

setter 只是把"赋给 computed 的值"转写到上游。如：

```typescript
const fullName = computed({
  get: () => `${firstName.value} ${lastName.value}`,
  set: (v) => {
    const [f, l] = v.split(' ')
    firstName.value = f         // ← 改了上游
    lastName.value  = l
  }
})

fullName.value = 'Jerry Smith'
// 走 setter → 改 firstName / lastName
// firstName 改了 → 触发 fullName.effect scheduler
// → _dirty=true → 下游重新读时自动重算
```

注意 setter 本身**不缓存任何东西**，所有缓存的失效都是通过上游的响应式触发完成的。

#### 6. 与 `methods` / `watch` / `ref` 的区别

| 维度 | `computed` | `methods` | `watch` | `ref` |
|------|-----------|-----------|---------|--------|
| 缓存 | ✅ 依赖不变就不重算 | ❌ 每次调用都执行 | — | ❌ |
| 调用方式 | 当属性读 `.value` | 当函数 `fn()` | 不调用，是副作用 | 当属性读 `.value` |
| 是否惰性 | ✅ 不读不算 | — | ❌ 创建即注册 | — |
| 返回值 | 必须有 | 可有可无 | 不关心 | 任意值 |
| 异步 | ❌ 不应有副作用 | ✅ | ✅ 专门处理副作用 | — |
| 适用场景 | 派生数据 | 一次性动作 | 数据变化时的副作用 | 单值容器 |

#### 7. 常见坑

##### ① 在 computed 里写副作用

```typescript
// ❌ 反模式
const list = computed(() => {
  fetch('/api/data')        // 副作用！多次读 .value 时机不可控
  return rawList.value
})

// ✅ 应该用 watchEffect 或 watch
watchEffect(() => fetch('/api/data?id=' + id.value))
```

##### ② 修改只读 computed 报警告

```typescript
const c = computed(() => count.value * 2)
c.value = 10   // ⚠️ warn: Write operation failed: computed value is readonly
```

只读 computed 的 setter 在源码中是 `NOOP`，仅在 dev 模式 warn。

##### ③ 在 computed 里改自己依赖的值（死循环）

```typescript
// ❌ 死循环
const c = computed(() => {
  count.value++         // 改了依赖
  return count.value
})
```

依赖变 → scheduler 标 dirty → 下游访问 → 再次执行 getter → 又改 → ...

##### ④ 解构丢失响应性

```typescript
const c = computed(() => count.value * 2)

const { value } = c      // value 是快照，后续不会更新
// 想保留响应性应该传 c 本身，或使用 toRef
```

##### ⑤ 在 computed 中依赖未追踪的数据

```typescript
let mode = 'a'                          // ❌ 普通变量，不是响应式
const c = computed(() => mode === 'a' ? x.value : y.value)
mode = 'b'                              // c 不会更新！
```

`mode` 不是响应式数据，改它不会触发 scheduler。要么用 ref，要么承认它在生命周期内不变。

##### ⑥ 自引用读 `.value`（罕见但存在）

```typescript
const c = computed(() => c.value + 1)   // 自我引用 → 死循环爆栈
```

源码里有 ReactiveEffect.run 的 parent 链检查可以兜底（避免同一 effect 嵌套递归），但仍然是不应写的反模式。

---

#### 8. 一句话总结

> `computed` 把 getter 包装成一个带 **scheduler 的 ReactiveEffect**：
> - **创建**时只建关系，不算
> - **首读**时跑一次 getter 并缓存，同时把当前 effect 收为下游
> - **依赖变化**时 scheduler 只标 `_dirty=true` 并通知下游，**不重算**
> - **下游再次读取**时才真正重算
>
> 这就是 **lazy + dirty** 双缓存：**不读不算，依赖未变不算，多次依赖变化合并成一次算**。

### 五、`IS_REACTIVE` 标志是怎么"加"到对象上的？—— Proxy getter 伪造

#### 源码片段（`packages/reactivity/src/reactive.ts`）

```typescript
export function isReactive(value: unknown): boolean {
  if (isReadonly(value)) {
    return isReactive((value as Target)[ReactiveFlags.RAW])
  }
  return !!(value && (value as Target)[ReactiveFlags.IS_REACTIVE])
}
```

#### 核心问题

> **`IS_REACTIVE` 是在哪里、什么时候被加到对象上的？**

#### 简短答案

> **`IS_REACTIVE` 根本没有"加"到任何对象上**。它是 Proxy 的 getter 拦截器**伪造**出来的一个返回值。

#### 1. `ReactiveFlags` 只是字符串常量

```typescript
export const enum ReactiveFlags {
  SKIP        = '__v_skip',
  IS_REACTIVE = '__v_isReactive',
  IS_READONLY = '__v_isReadonly',
  IS_SHALLOW  = '__v_isShallow',
  RAW         = '__v_raw'
}
```

`ReactiveFlags.IS_REACTIVE` === `'__v_isReactive'`。

#### 2. 真相在 createGetter 里

```typescript
function createGetter(isReadonly = false, shallow = false) {
  return function get(target: Target, key: string | symbol, receiver: object) {
    if (key === ReactiveFlags.IS_REACTIVE) {
      return !isReadonly                  // ★ 不查 target，直接返回闭包里的标记
    } else if (key === ReactiveFlags.IS_READONLY) {
      return isReadonly                   // ★ 同上
    } else if (key === ReactiveFlags.IS_SHALLOW) {
      return shallow                      // ★ 同上
    }
    // ... 之后才是 Reflect.get(target, key, receiver)
  }
}
```

访问 `state.__v_isReactive` 时，Proxy 的 get handler 拦截下来，**直接返回闭包变量 `!isReadonly`**——根本不会去读原始对象上的属性。

#### 3. 完整运行链路

```javascript
const raw = { a: 1 }
const state = reactive(raw)

state.__v_isReactive
//        │
//        ▼
//  ① 触发 Proxy 的 get handler
//        │ target = raw, key = '__v_isReactive'
//        ▼
//  ② if (key === ReactiveFlags.IS_REACTIVE) return !isReadonly
//        │
//        ▼
//  ③ 直接返回 true（reactive 创建时 isReadonly=false）
//
// ★ 全过程 raw 上根本没有 __v_isReactive 这个属性
console.log(raw.__v_isReactive)   // undefined
```

证明：

```javascript
Object.keys(raw)               // ['a']            ← 没有 __v_isReactive
Object.keys(state)             // ['a']            ← Proxy 也透传，没有 __v_isReactive
'__v_isReactive' in raw        // false
raw['__v_isReactive']          // undefined
state['__v_isReactive']        // true ← 由 getter 伪造
```

#### 4. `isReadonly` / `shallow` 这些标记是哪儿来的？

是 `createGetter` 的**闭包参数**，在创建 handler 那一刻就固化了：

```typescript
// packages/reactivity/src/baseHandlers.ts
export const mutableHandlers          = { get: createGetter() }                 // false, false
export const readonlyHandlers         = { get: createGetter(true) }             // true,  false
export const shallowReactiveHandlers  = { get: createGetter(false, true) }      // false, true
export const shallowReadonlyHandlers  = { get: createGetter(true,  true) }      // true,  true
```

- `reactive(obj)` 用 `mutableHandlers` 创建 Proxy
- `readonly(obj)` 用 `readonlyHandlers`
- `shallowReactive(obj)` 用 `shallowReactiveHandlers`
- ...

**handler 是哪种**就决定了 `IS_REACTIVE` / `IS_READONLY` / `IS_SHALLOW` 各自返回什么。

#### 5. 为什么这样设计，而不是真的写到对象上？

| 方案 | 缺点 |
|------|------|
| 在原始对象上加 `__v_isReactive = true` | 污染用户对象；`JSON.stringify` 暴露这些字段；遍历会出现；冻结/只读对象无法加 |
| Proxy getter 拦截伪造 | ✅ 不动用户原始对象；✅ 同一原始对象可同时是 reactive 也可被 readonly 再包一层，互不冲突；✅ `Object.keys` 干净 |

特别地——同一个 raw 可以同时存在于多种 Proxy 中，互不干扰：

```javascript
const raw = { a: 1 }
const r   = reactive(raw)
const ro  = readonly(raw)

isReactive(r)   // true  ← r 的 getter 返回 !false = true
isReactive(ro)  // false ← ro 的 getter 因 isReadonly=true 返回 !true = false
isReadonly(ro)  // true
```

如果是"写在对象上"的方案，这种"同一原始数据多重身份"根本无法实现。

#### 6. `isReactive` 为什么要穿透 readonly？

```typescript
export function isReactive(value: unknown): boolean {
  if (isReadonly(value)) {
    return isReactive((value as Target)[ReactiveFlags.RAW])  // ★ 递归
  }
  return !!(value && (value as Target)[ReactiveFlags.IS_REACTIVE])
}
```

考虑这种嵌套：

```javascript
const r  = reactive({ a: 1 })   // r 是 reactive
const ro = readonly(r)          // ro 包裹 r：外层 readonly + 内层 reactive

isReactive(ro)
// ① isReadonly(ro) → true
// ② 访问 ro[RAW]   → ro 的 getter 命中 RAW 分支，返回 r（被包的原始对象）
// ③ 递归 isReactive(r) → true ✅
```

`RAW` 分支也是在 getter 里：

```typescript
else if (
  key === ReactiveFlags.RAW &&
  receiver === (isReadonly ? readonlyMap : reactiveMap).get(target)
) {
  return target   // ← 返回被代理的原始对象
}
```

#### 7. 跟 `__v_isRef` 的对比

| 标志 | 添加方式 | 真实存储在哪 | `Object.keys` 能看到吗 |
|------|----------|--------------|------------------------|
| `__v_isRef` | `RefImpl` 类**显式声明**的实例字段 | 实例对象上 | ✅ 能（取决于可枚举性） |
| `__v_isReactive` | Proxy getter **拦截伪造** | 不存在任何对象上 | ❌ 不能 |

**为什么不一致？**
- Ref 是 Vue 自己造的对象（`RefImpl` 实例），可以随便加字段
- Reactive 必须基于**用户传入的原始对象**包 Proxy，加字段会污染用户数据

#### 8. 一句话总结

> `IS_REACTIVE` 不是被"加"到对象上的，而是 Proxy 的 getter handler 在闭包里记住了 `isReadonly` 标记，当访问 `__v_isReactive` 这个特殊 key 时**直接返回** `!isReadonly`。
> 整个机制是"**handler 即身份**"：是哪种 handler 创建的 Proxy，就报告自己是哪种响应式对象。

### 六、`watch` 的 reactive 分支为什么能收集到依赖？—— traverse 强制深度访问

#### 源码片段（`packages/runtime-core/src/apiWatch.ts`）

```typescript
if (isRef(source)) {
  getter = () => source.value
  forceTrigger = isShallow(source)
} else if (isReactive(source)) {
  getter = () => source       // ① 只是返回 Proxy 引用？！
  deep = true                 // ② 强制开启 deep
}

// ...

if (cb && deep) {
  const baseGetter = getter
  getter = () => traverse(baseGetter())   // ★ 二次包装
}
```

#### 核心问题

> `getter = () => source` 看起来只是返回 source 引用本身，并没访问任何属性，**为什么 watch 能收集到依赖、深层修改也能触发回调？**

#### 1. 单看 `() => source` 确实不会收集依赖

```typescript
const state = reactive({ a: { b: 1 } })

const getter = () => state

effect(() => {
  const v = getter()   // 拿到 Proxy 引用，没访问任何属性
  // ❌ 此处没有任何 track 调用，依赖表是空的
})

state.a.b = 2          // ❌ 永远不会触发回调
```

**只有真的访问到属性，Proxy 的 get 才会触发 `track`**，单纯返回 Proxy 引用是不够的。

#### 2. 秘密在 `deep = true` 配合的二次包装

注意 `isReactive(source)` 分支后面紧跟着 `deep = true`，让后面这段生效：

```typescript
if (cb && deep) {
  const baseGetter = getter
  getter = () => traverse(baseGetter())   // ← 真正生效的 getter
}
```

所以最终的 getter 是 `() => traverse(source)`。**真正收集依赖的是 `traverse`**。

#### 3. `traverse` 的实现：递归访问每一层属性

```typescript
export function traverse(value: unknown, seen?: Set<unknown>) {
  if (!isObject(value) || (value as any)[ReactiveFlags.SKIP]) {
    return value
  }
  seen = seen || new Set()
  if (seen.has(value)) {
    return value           // 防循环引用
  }
  seen.add(value)

  if (isRef(value)) {
    traverse(value.value, seen)
  } else if (isArray(value)) {
    for (let i = 0; i < value.length; i++) {
      traverse(value[i], seen)              // ← 访问 value[i] 触发 get
    }
  } else if (isSet(value) || isMap(value)) {
    value.forEach((v: any) => traverse(v, seen))
  } else if (isPlainObject(value)) {
    for (const key in value) {
      traverse((value as any)[key], seen)   // ← 访问 value[key] 触发 get
    }
  }
  return value
}
```

关键就在 `for (const key in value) traverse(value[key], seen)`：

1. `value` 是 reactive 出来的 Proxy
2. 读 `value[key]` 命中 Proxy 的 **get 拦截器**
3. get 拦截器里调用 `track(target, GET, key)` —— **依赖被收集！**
4. 拿到的值若仍是 Proxy（嵌套对象），递归 `traverse` 再访问下一层 —— **深层依赖也被收集！**

#### 4. 完整链路

```
new ReactiveEffect(() => traverse(source), scheduler)
        │
        ▼
   effect.run()
        │
        ├─ activeEffect = this              ← 标记当前活跃 effect
        ├─ 执行 getter() = traverse(source)
        │       │
        │       └─ 遍历访问 source 每个属性
        │             │
        │             └─ 触发 Proxy.get
        │                   │
        │                   └─ track(target, key)
        │                         │
        │                         └─ 把 activeEffect 加入 dep ✅
        │                              （所有嵌套属性都被收集）
        │
        └─ 返回 source 作为 oldValue
```

#### 5. 为什么 reactive 分支要**强制** `deep = true`？

`watch(reactiveObj, cb)` 的语义是：**这个对象（及其任意深度的内部属性）变了，我都要收到通知**。

但 Proxy **只能拦截被访问到的层级**——不主动深度访问，就只能拿到顶层引用，无法收集深层依赖。所以 Vue 强制：

> `watch(reactiveObject, cb)` ⟹ 等价于 `watch(reactiveObject, cb, { deep: true })`

用户即使没传 `deep`，源码也帮他设上。

#### 6. 与 `isRef` 分支的对比

| 分支 | getter | 是否强制 deep | 原因 |
|------|--------|--------------|------|
| `isRef(source)` | `() => source.value` | ❌ 用户决定 | 访问 `.value` 已触发 RefImpl 的 track，浅层依赖足够 |
| `isReactive(source)` | `() => source` + `deep=true` | ✅ 强制 | 没有 `.value` 入口，只能用 traverse 模拟"读一遍" |
| `isFunction(source)` | 用户的 getter 函数 | ❌ 用户决定 | 用户在 getter 里读什么就收集什么，控制权在用户手上 |

#### 7. 小心：`watch(reactiveObj, cb)` 拿到的 oldValue 和 newValue 是同一个引用

```typescript
const state = reactive({ count: 0 })
watch(state, (newVal, oldVal) => {
  console.log(newVal === oldVal)   // true ！！
  console.log(newVal === state)    // true
})
state.count++
```

因为 getter 总是返回 `source` 本身（同一个 Proxy），所以 newValue 和 oldValue 是**同一对象**，回调里**无法通过比较 newVal/oldVal 拿到旧值**。这是 reactive 直接 watch 的限制——想拿到具体变化前的值要用 getter 形式：

```typescript
watch(() => state.count, (n, o) => { /* n=1, o=0 */ })
```

#### 8. 一句话总结

> `getter = () => source` 自身不收集依赖。关键在紧跟的 `deep = true` 让后续逻辑把 getter 包成 `() => traverse(source)`。
> `traverse` 通过 `for...in` 递归读取每一层属性，每次读都触发 Proxy 的 get 拦截器调用 `track`——这才完成了"watch 一个 reactive 对象就能侦听其全部深层变化"的能力。
