### 一、JavaScript 进阶手写题

#### 1. 深拷贝（含循环引用、Map/Set/Symbol）

```javascript
/**
 * 深拷贝 —— 完整版
 *
 * 关键点：
 *   ① 递归拷贝所有嵌套对象
 *   ② 用 WeakMap 记录"已拷贝过的对象"解决循环引用
 *   ③ 特殊类型分别处理：Date / RegExp / Map / Set / Symbol / Function
 *   ④ 不可枚举属性 & Symbol key 用 Reflect.ownKeys 捕获
 */
function deepClone(obj, hash = new WeakMap()) {
  // null 和 非对象直接返回
  if (obj === null || typeof obj !== 'object') return obj

  // 循环引用：已经拷贝过就直接返回缓存
  if (hash.has(obj)) return hash.get(obj)

  // ============ 特殊内置类型：按值语义重建 ============

  // ① Date —— 日期对象
  //    Date 内部存的是一个时间戳（getTime() 毫秒数）
  //    new Date(oldDate) 会调用 oldDate.valueOf() 拿到时间戳，等价于 new Date(oldDate.getTime())
  //    → 这样新旧 Date 完全独立，修改新 Date 不影响旧 Date
  if (obj instanceof Date) return new Date(obj)

  // ② RegExp —— 正则对象
  //    正则有两个关键属性：source（正则表达式正文）和 flags（修饰符 g/i/m/s/u/y）
  //    直接 new RegExp(oldRegExp) 也有效，但显式拆 source + flags 更清晰
  //    lastIndex（exec/test 的位置记录）不需要拷贝——它属于运行时状态，不是对象"值"的一部分
  if (obj instanceof RegExp) return new RegExp(obj.source, obj.flags)

  // ============ 可迭代集合：Map / Set ============

  // ③ Map —— 键值对集合
  //    为什么不能直接 new Map(oldMap)？因为 oldMap 的 key 和 value 都可能是引用类型
  //    必须递归深拷贝 key 和 value，否则新旧 Map 会共享内部的对象引用
  if (obj instanceof Map) {
    const copy = new Map()                          // 空 Map 容器
    hash.set(obj, copy)                             // ☆ 先登记缓存（重要！防止 Map 内部引用自己导致死循环）
    obj.forEach((v, k) => {
      //        ↑ forEach 的参数顺序是 (value, key)，不是 (key, value)——这是 Map 的 API 设计
      copy.set(
        deepClone(k, hash),    // key 也可能是对象（虽然少见，但 Map 允许对象当 key）
        deepClone(v, hash)     // value 递归深拷贝
      )
    })
    return copy
  }

  // ④ Set —— 不重复值的集合
  //    同理，Set 内的元素可能是引用类型，必须逐个深拷贝
  if (obj instanceof Set) {
    const copy = new Set()                          // 空 Set 容器
    hash.set(obj, copy)                             // ☆ 先登记缓存
    obj.forEach(v => {
      copy.add(deepClone(v, hash))                  // 每个元素递归深拷贝
    })
    return copy
  }

  // ============ 为什么没有 WeakMap / WeakSet？ ============

  // ★ WeakMap 和 WeakSet 不能被深拷贝，原因有三层：
  //
  // ① 不可遍历（致命）：
  //    WeakMap/WeakSet 没有 forEach、keys()、values()、entries() 方法，
  //    也不可迭代（没有 [Symbol.iterator]）。这是它们的核心设计——
  //    key 是弱引用，随时可能被 GC 回收，如果允许遍历，你就能"看到"一个
  //    已经被回收的 key 的痕迹，这在语义上自相矛盾。
  //    所以 WeakMap 上调用 obj.forEach → TypeError: obj.forEach is not a function。
  //
  // ② 即使能遍历，拷贝也毫无意义：
  //    WeakMap 的 key 必须是对象，且只被弱引用。
  //    假设你能拷贝，拷贝出来的新对象 key 在新 WeakMap 中不再有外部强引用
  //    → 立即被 GC 回收 → 新 WeakMap 变成空的。
  //
  // ③ 弱引用的"快照"语义不存在：
  //    WeakMap 的值在 GC 前后是不同的集合——你拷贝"当前可见的条目"，
  //    下一秒就可能因 GC 而改变。深拷贝追求的是"值等价"，而 WeakMap
  //    的内容本质上是非确定性的，不存在稳定等价。
  //
  // → 遇到 WeakMap/WeakSet，要么跳过，要么直接抛错。

  // ============ 普通对象 / 数组 ============

  // ⑤ 数组：用 [] 创建空数组
  //    用 Array.isArray 而不是 obj.constructor === Array
  //    因为 obj 可能来自 iframe/cross-realm，其 constructor 是 iframe 内的 Array（见 instanceof 章节）
  // ⑥ 普通对象：用 Object.create(proto) 创建，保留原型链
  //    为什么不用 {}？因为 {} 的原型是 Object.prototype，会丢失 obj 原本的原型链
  //    例：class Dog extends Animal {} → new Dog() 的原型是 Dog.prototype → Animal.prototype
  //    用 {} 拷贝会变成 Object.prototype，原型链断裂 → 原型方法丢失
  const copy = Array.isArray(obj)
    ? []
    : Object.create(Object.getPrototypeOf(obj))

  // ☆ 关键步骤：先把 copy 注册到 WeakMap
  //    必须在遍历属性之前做，否则当 obj 的属性引用了 obj 自身（或祖先对象）时，
  //    递归 deepClone 又会进入这个 obj → 形成死循环 / 爆栈
  //    WeakMap 在这里充当"已拷贝对象索引"——递归前查一下，已拷贝的直接返回
  hash.set(obj, copy)

  // ☆ Reflect.ownKeys vs Object.keys 的区别：
  //    Object.keys(obj)       → 只拿到可枚举的字符串属性
  //    Reflect.ownKeys(obj)   → 可枚举 + 不可枚举 + Symbol key ★ 全覆盖
  //    例：const s = Symbol('id'); obj[s] = 1;
  //    Object.keys 拿不到 s；Reflect.ownKeys 可以
  Reflect.ownKeys(obj).forEach(key => {
    // 逐属性递归深拷贝，把 hash 传递下去以处理循环引用
    copy[key] = deepClone(obj[key], hash)
  })

  return copy
}
```

**面试追问**：为什么用 `WeakMap` 而不是 `Map`？

→ 关键在于**强引用 vs 弱引用**的差别：

```javascript
// ====== Map：强引用 ======
const hash = new Map()
let obj = { name: 'Tom' }
hash.set(obj, '已拷贝')   // Map 对 obj 持有强引用
obj = null                 // 你清掉了自己的引用...

// obj 仍不会被 GC 回收！
// 因为 Map 还"死死拽着" obj，GC 看到有强引用就不敢回收 → 内存泄漏
```

```javascript
// ====== WeakMap：弱引用 ======
const hash = new WeakMap()
let obj = { name: 'Tom' }
hash.set(obj, '已拷贝')   // WeakMap 对 obj 只有弱引用——"我引用你，但不阻止你被回收"
obj = null                 // 现在世界上没有任何"强引用"指向 {name:'Tom'} 了

// ↓ 下一次 GC 运行时：
// {name:'Tom'} 被回收 → WeakMap 中对应的记录自动消失 → 没有内存泄漏
```

**回到深拷贝场景**：

`hash` 存的是 `原对象 → 拷贝结果` 的映射。用 `Map` 的问题：

```
deepClone(hugeObj) 执行完毕
  → 函数返回，你不再需要 hugeObj
  → 但 hash（Map）还死死抓着 hugeObj 的引用
  → GC 不敢回收 hugeObj
  → hugeObj 占着内存不放 → 这就是内存泄漏
```

用 `WeakMap` 则不会：函数执行完 → `hash` 随着栈帧消失 → 原对象不再有强引用 → GC 正常回收。

**一句话**：`WeakMap` 让深拷贝的"已拷贝索引"不产生副作用——拷贝完了就放手，不阻止原对象被 GC。

#### 2. 防抖 & 节流（含立即执行）

```javascript
// ====== 防抖 Debounce ======
// 场景：搜索输入框联想、窗口 resize
// 思路：每次触发都重置定时器，直到"安静"了 wait ms 才执行
function debounce(fn, wait = 300, immediate = false) {
  let timer = null
  return function (...args) {
    const callNow = immediate && !timer  // ★ 首次触发立即执行

    clearTimeout(timer)
    timer = setTimeout(() => {
      timer = null
      if (!immediate) fn.apply(this, args)
    }, wait)

    if (callNow) fn.apply(this, args)
  }
}

// ====== 节流 Throttle ======
// 场景：滚动监听、拖拽、鼠标移动
// 思路：在 wait ms 内最多执行一次
function throttle(fn, wait = 300) {
  let lastTime = 0
  return function (...args) {
    const now = Date.now()
    if (now - lastTime >= wait) {
      lastTime = now
      fn.apply(this, args)
    }
  }
}
```

#### 3. `Promise.all` / `Promise.allSettled` / `Promise.race` / `Promise.any`

```javascript
// ====== Promise.all ======
// 全部成功 → 结果数组；任意一个失败 → 立即 reject
Promise.myAll = function (promises) {
  return new Promise((resolve, reject) => {
    if (!promises.length) return resolve([])
    const results = []
    let count = 0
    promises.forEach((p, i) => {
      Promise.resolve(p).then(
        v => {
          results[i] = v            // ★ 用下标保持顺序
          count++
          if (count === promises.length) resolve(results)
        },
        reject                      // 一个失败 → 直接 reject
      )
    })
  })
}

// ====== Promise.allSettled ======
// 全部 settled（无论成功/失败）→ 结果数组 [{status,value/reason},...]
Promise.myAllSettled = function (promises) {
  return new Promise(resolve => {
    if (!promises.length) return resolve([])
    const results = []
    let count = 0
    promises.forEach((p, i) => {
      Promise.resolve(p).then(
        v  => { results[i] = { status: 'fulfilled', value: v };  count++; if (count === promises.length) resolve(results) },
        e  => { results[i] = { status: 'rejected',  reason: e }; count++; if (count === promises.length) resolve(results) }
      )
    })
  })
}

// ====== Promise.race ======
// 第一个 settled 的结果（无论成功/失败）
Promise.myRace = function (promises) {
  return new Promise((resolve, reject) => {
    promises.forEach(p => Promise.resolve(p).then(resolve, reject))
  })
}

// ====== Promise.any ======
// 第一个 fulfilled 的结果；全部失败 → AggregateError
Promise.myAny = function (promises) {
  return new Promise((resolve, reject) => {
    if (!promises.length) return reject(new AggregateError([], 'All promises were rejected'))
    let count = 0
    promises.forEach(p => {
      Promise.resolve(p).then(resolve, () => {
        count++
        if (count === promises.length) reject(new AggregateError('All promises were rejected'))
      })
    })
  })
}
```

#### 4. 并发限制的 Promise 调度器

```javascript
/**
 * 场景：有 100 个请求，但需要限制同时最多 3 个并发
 * 思路：维护一个执行池，每完成一个就从队列取下一个补上
 */
function asyncPool(limit, tasks) {
  const results = []
  const executing = []          // 正在执行中的 Promise

  for (const [i, task] of tasks.entries()) {
    // 包装一层：记录结果 + 执行完从 executing 中移除
    const p = Promise.resolve(task()).then(v => {
      results[i] = { status: 'fulfilled', value: v }
    }, e => {
      results[i] = { status: 'rejected', reason: e }
    }).finally(() => {
      executing.splice(executing.indexOf(p), 1)  // 执行完，从池中移出
    })

    executing.push(p)

    if (executing.length >= limit) {
      // 池满了，等最快完成的那个
      await Promise.race(executing)
    }
  }

  // 等最后一批全部完成
  await Promise.all(executing)
  return results
}
```

#### 5. 发布-订阅模式（EventEmitter）

```javascript
class EventEmitter {
  constructor() {
    this.events = {}   // { eventName: [callback1, callback2, ...] }
  }

  on(event, cb) {
    (this.events[event] ||= []).push(cb)    // 订阅
  }

  off(event, cb) {
    if (!this.events[event]) return
    this.events[event] = this.events[event].filter(fn => fn !== cb)  // 取消订阅
  }

  once(event, cb) {
    const wrapper = (...args) => { cb(...args); this.off(event, wrapper) }
    this.on(event, wrapper)                 // 执行一次后自动解绑
  }

  emit(event, ...args) {
    (this.events[event] || []).forEach(cb => cb(...args))  // 发布
  }
}
```

#### 8. 柯里化（Currying）

```javascript
// 把 f(a, b, c) 变成 f(a)(b)(c)
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) return fn.apply(this, args)   // 参数够了，执行
    return (...args2) => curried.apply(this, args.concat(args2)) // 继续收集
  }
}
// 调用：const add = curry((a, b, c) => a + b + c);  add(1)(2)(3) → 6
```

**柯里化的核心价值**：把"一次性决定所有参数"变成"分批传参、分步执行"——先确定一部分，剩下的后面再说。

实际场景不是 `add(1)(2)(3)` 这种逐个传的写法，而是**先固定复用度高的参数，生成一个配好的新函数**：

```javascript
// ====== 场景 1：日志工具 ======
const log = curry((level, module, message) => {
  console.log(`[${level}] [${module}] ${message}`)
})

// 把 level + module 固定好，生产"这个模块的日志函数"
const warnAuth = log('WARN', 'auth')     // a='WARN' b='auth' 已固定，只需传 message
const errorDB  = log('ERROR', 'database')

warnAuth('登录失败次数超限')              // [WARN] [auth] 登录失败次数超限
errorDB('连接超时')                       // [ERROR] [database] 连接超时

// 对比不用 curry：
log('WARN', 'auth', '登录失败')          // 每次都要写全三个参数——重复且易拼错

// ====== 场景 2：DOM 事件绑定 ======
const on = curry((event, selector, handler) => {
  document.querySelector(selector)?.addEventListener(event, handler)
})

const onClick = on('click')              // 固定事件类型

onClick('#save-btn', () => save())       // 只传 selector + handler
onClick('#delete-btn', () => remove())

// ====== 场景 3：数据格式化 ======
const formatNumber = curry((locale, currency, n) => {
  return new Intl.NumberFormat(locale, { style: 'currency', currency }).format(n)
})

const formatCNY = formatNumber('zh-CN', 'CNY')
const formatUSD = formatNumber('en-US', 'USD')

formatCNY(1234.56)   // "¥1,234.56"
formatCNY(7890.12)   // "¥7,890.12"    ← 同一配置复用多次
formatUSD(99.99)     // "$99.99"
```

> **一句话**：柯里化不是让你写 `f(a)(b)(c)`，而是让你轻松造出 `const fWithFixedArg = f(fixedArg)`——把"通用函数"变成一堆"配置好的专用函数"，减少重复传参。这是函数式编程中 compose / pipe 组合的基础设施。

#### 9. LRU 缓存

```javascript
/**
 * 最近最少使用缓存，超出容量时淘汰最久未使用的
 * 用 Map 实现（Map 的 key 按插入顺序排列，支持顺序遍历）
 */
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity
    this.cache = new Map()
  }

  get(key) {
    if (!this.cache.has(key)) return -1
    const value = this.cache.get(key)
    this.cache.delete(key)          // ★ 删除后重新插入 → 排到"最新"
    this.cache.set(key, value)
    return value
  }

  put(key, value) {
    if (this.cache.has(key)) this.cache.delete(key)  // 更新也移到最新
    this.cache.set(key, value)
    if (this.cache.size > this.capacity) {
      // Map 的第一个 key 就是最久未使用的
      this.cache.delete(this.cache.keys().next().value)
    }
  }
}
```

### 二、浏览器 & HTTP 高频题

#### 1. 从输入 URL 到页面渲染（必考题）

```
① DNS 解析：域名 → IP 地址（递归 + 迭代查询，有浏览器/系统/hosts/路由器缓存）
② TCP 三次握手：SYN → SYN+ACK → ACK（建立可靠连接）
③ TLS 握手（HTTPS）：ClientHello → ServerHello → 证书验证 → 密钥交换
④ 发送 HTTP 请求：请求行 + 请求头 + 请求体
⑤ 服务器处理并返回 HTTP 响应：状态行 + 响应头 + 响应体
⑥ 浏览器解析 HTML → 构建 DOM 树
⑦ 解析 CSS → 构建 CSSOM 树
⑧ DOM + CSSOM → Render Tree（渲染树）
⑨ Layout（回流/重排）：计算每个节点的几何位置和大小
⑩ Paint（重绘）：将像素绘制到屏幕
⑪ Composite（合成）：GPU 将各图层合并显示
```

#### 2. HTTP 缓存策略

```
       浏览器请求
           │
     ┌─────▼──────┐
     │  有缓存吗？  │─── NO ──→ 向服务器发起请求
     └─────┬──────┘
           │ YES
     ┌─────▼──────────┐
     │ 强缓存命中？     │  ← Cache-Control: max-age / Expires
     │ (不发请求)      │
     └─────┬──────────┘
      YES  │         NO
       │   │          │
       │   │    ┌─────▼──────────┐
       │   │    │ 协商缓存？      │  ← ETag/If-None-Match
       │   │    │ (发请求带标识)  │    Last-Modified/If-Modified-Since
       │   │    └─────┬──────────┘
       │   │    304  │        200
       │   │    (用缓存)    (返回新内容)
       ▼   ▼       ▼         ▼
        直接使用缓存         更新缓存
```

#### 3. HTTP/1.1 → HTTP/2 → HTTP/3

| 版本 | 核心特性 | 痛点 |
|------|---------|------|
| HTTP/1.1 | keep-alive 复用连接、管道化 | 队头阻塞（HOL）、每个请求一个 TCP 连接 |
| HTTP/2 | 二进制分帧、多路复用、头部压缩 HPACK、Server Push | TCP 层的队头阻塞仍在 |
| HTTP/3 | 基于 QUIC（UDP）、0-RTT 建连、无 TCP HOL | 普及率待提高 |

#### 4. HTTPS 握手原理

```
1. Client Hello → 支持的加密套件 + 客户端随机数
2. Server Hello → 选定加密套件 + 服务器随机数 + 数字证书（含公钥）
3. 客户端验证证书 → CA 公钥解密签名，确认身份
4. 客户端生成 Pre-Master Key，用服务器公钥加密发给服务器
5. 双方用 客户端随机数 + 服务器随机数 + Pre-Master Key → 生成会话密钥
6. Finished 消息：后续数据用会话密钥（对称加密）传输

关键：非对称加密仅用于交换密钥（慢但安全），数据传输用对称加密（快）
```

#### 5. 跨域解决方案

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **CORS**（主流） | 服务器设置 `Access-Control-Allow-Origin` | 有服务器控制权 |
| **反向代理** | 同源对浏览器透明，代理转发到异源 | 开发环境（Vite/Webpack devServer） |
| **JSONP** | 动态创建 `<script>` 标签，回调函数 | 仅 GET，已淘汰 |
| **postMessage** | iframe/窗口间消息通信 | 跨窗口通信 |
| **WebSocket** | 不受同源策略限制 | 双向实时通信 |
| **Nginx 反向代理** | 同域下转发不同服务 | 生产环境 |

**Nginx 反向代理详解**：这是生产环境最常用的跨域方案。原理是在同域名下配置 Nginx，将不同路径的请求转发（`proxy_pass`）到不同的后端服务。对浏览器来说，所有请求都是发给同一个域——**跨域从根上就不存在了**。

```
浏览器                          Nginx (port 80)                  后端服务
──────                          ────────────────                 ────────

请求 /api/user
  →  http://example.com/api/user
      同域，不跨域 ✓
                                 匹配 location /api/ →
                                 proxy_pass http://api-server:3001
                                                                   处理请求
                                                                   返回数据
                                  ← 响应返回给浏览器

请求 /static/logo.png
  →  http://example.com/static/logo.png
      同域，不跨域 ✓
                                 匹配 location /static/ →
                                 proxy_pass http://cdn-server:8080
                                                                   返回文件
                                  ← 响应返回给浏览器
```

```nginx
# nginx.conf 典型配置

server {
    listen       80;
    server_name  example.com;

    # 前端静态资源
    location / {
        root   /var/www/html;          # 前端打包产物，如 dist/
        index  index.html;
        try_files $uri /index.html;    # SPA 路由 fallback
    }

    # ★ 反向代理：/api 开头的请求转发到后端 API 服务
    location /api/ {
        proxy_pass http://127.0.0.1:3001/;    # 后端 API 地址
        proxy_set_header Host $host;           # 透传原始域名
        proxy_set_header X-Real-IP $remote_addr;  # 透传客户端真实 IP
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**关键点**：

- 浏览器只跟一个域打交道（`example.com`），所有请求都是同域——不存在跨域
- Nginx 在服务端做转发，服务端没有同源策略限制
- 与开发环境 Vite/Webpack devServer `proxy` 配置完全同理，只不过前者运行在 Nginx 上、后者跑在 devServer 进程中
- 优势：不需要后端改 CORS 头、不需要 JSONP、不受浏览器兼容性影响、支持所有 HTTP 方法
- 请求链路：`浏览器 → Nginx（同域入口） → 转发到各后端服务 → 响应原路返回`

### 三、Vue 面试高频题

#### 1. Vue2 `Object.defineProperty` vs Vue3 `Proxy`

| 维度 | Vue2（defineProperty） | Vue3（Proxy） |
|------|----------------------|---------------|
| 监听方式 | 递归遍历每个属性 + getter/setter | 代理整个对象 |
| 新增/删除属性 | **监听不到**（需 `$set`/`$delete`） | ✅ 自动监听 |
| 数组索引修改 | ❌ 监听不到 | ✅ 自动监听 |
| 数组 length | ❌ 监听不到 | ✅ 自动监听 |
| Map/Set | ❌ 不支持 | ✅ 支持 |
| 性能 | 初始化时递归遍历所有属性（慢） | 惰性代理，用到才递归（快） |
| 兼容性 | 支持 IE | 不支持 IE |

#### 2. `ref` vs `reactive`

```javascript
// ref：包装基本类型，通过 .value 访问/修改
const count = ref(0)
count.value++                        // 模板中自动解包，不需要 .value

// reactive：包装对象，直接访问属性
const state = reactive({ count: 0 })
state.count++

// 核心区别：
//   ref 内部也是用 reactive 实现的（对对象类型）
//   reactive 不能重新赋值（会丢失响应式），ref 可以
//
//   ★ 解构 reactive 的"原始值属性"会丢失响应式：
//      const { count } = reactive({ count: 0, user: { name: 'Tom' } })
//      count  → 0（普通数字，丢失响应式连接）
//      user   → Proxy{ name:'Tom' }（仍是 reactive 对象，响应式还在）
//      原因：reactive 是深度代理，嵌套对象自动被 Proxy 包裹——解构出来仍是 Proxy
//      但原始值（number/string/boolean）不能被 Proxy——解构后只是一个独立的普通值
//
//   想让每个属性都以独立的 ref 保持响应式连接 → 用 toRefs()
```

#### 3. `computed` vs `watch`

| | computed | watch |
|-------|---------|------|
| 缓存 | ✅ 依赖不变不会重新计算 | ❌ 无缓存 |
| 返回值 | 必须有返回值 | 不需要返回值 |
| 异步 | ❌ 不支持异步 | ✅ 支持异步操作 |
| 场景 | 派生状态（全名 = 姓 + 名） | 执行副作用（请求数据、操作 DOM） |
| 依赖追踪 | 自动追踪 | 需要明确指定 watch 的源 |

#### 4. `nextTick` 原理

```
this.count = 1            ← 触发 setter → 通知 watcher
this.count = 2            ← 再次触发 → watcher 被去重，只推入一次更新队列
this.count = 3            ← 同上

// Vue 内部将 DOM 更新推入微任务队列
// nextTick 的回调排在 DOM 更新之后执行

this.$nextTick(() => {
  console.log(this.$el.textContent)  // 拿到更新后的 DOM
})
```

本质：Vue 的 DOM 更新是**异步批处理**的，`nextTick` 让你在 DOM 更新完成后拿到最新 DOM。Vue3 优先使用 `Promise.then`（微任务），降级到 `setTimeout`（宏任务）。

#### 5. Vue3 Composition API 解决了什么问题

- **逻辑复用**：mixin 的命名冲突/来源不清晰 → composable 函数，来源可追踪
- **代码组织**：Options API 按选项分割（data 一堆、methods 一堆 → 分散）→ 按功能聚合
- **TypeScript 支持**：Composition API 天然友好

#### 6. Vue 的 diff 算法

Vue 的 diff 基于**同层比较 + 双端对比**：

```
1. 新旧虚拟 DOM 树的同层级节点比较（不同层级不比较，直接替换）
2. 双端对比算法（oldStart / oldEnd / newStart / newEnd 四个指针）：
   - 头头比较（相同 → 都后移）
   - 尾尾比较（相同 → 都前移）
   - 头尾比较（oldStart === newEnd → 节点移动）
   - 尾头比较（oldEnd === newStart → 节点移动）
   - 都不匹配 → 用 key 在旧节点中查找
3. 有 key 时按 key 复用，无 key 就地复用（index 作为 key 会出问题）
```

#### 7. Vue Router History vs Hash 模式

| | Hash 模式 | History 模式 |
|-------|---------|------------|
| URL 形式 | `/#/page` | `/page` |
| 原理 | `hashchange` 事件 | HTML5 `pushState`/`popState` |
| 服务器配置 | 不需要 | 需要配置 fallback 到 index.html |
| SEO | ❌ # 后内容不被搜索引擎收录 | ✅ 正常 |
| 兼容性 | 全部浏览器 | IE10+ |

### 四、React 面试高频题

#### 1. Hooks 为什么不能放在条件语句中

React 用**链表**存储 Hook 的状态。每次渲染时必须按**相同的顺序**调用 Hook，否则状态会错位：

```javascript
// ❌ 错误
if (condition) {
  useState(0)          // 第一次渲染调了，第二次没调 → 错位！
}
useState(1)            // 这个 Hook 读到了上一个状态的链表节点

// 本质上 React 内部是这样存储的：
// hook1 → hook2 → hook3  (按调用顺序形成的链表)
// 如果顺序变了，hook 对应的状态就对不上了
```

#### 2. `useEffect` vs `useLayoutEffect`

| | useEffect | useLayoutEffect |
|-------|---------|---------------|
| 执行时机 | 浏览器**绘制之后**异步执行 | DOM 更新后、**绘制之前**同步执行 |
| 是否会阻塞渲染 | 不会 | 会（同步执行完才绘制） |
| 场景 | 数据请求、订阅、日志 | 读取/修改 DOM 布局、避免闪烁 |

#### 3. `setState` 是同步还是异步

- **React 合成事件和生命周期**中：异步（批量更新）
- **setTimeout / 原生 DOM 事件**中：同步
- React 18 的 Automatic Batching 将批量更新扩展到所有场景

#### 4. Virtual DOM 真的快吗

Virtual DOM 不是为了"比直接操作 DOM 快"，而是为了：
- **减少不必要的 DOM 操作**（diff 出最小变更，批量更新）
- **跨平台渲染**（vdom 是一层抽象，可渲染到 DOM/Native/Canvas）
- **开发体验**（数据驱动，不用手动管理 DOM）

直接操作 DOM 通常比 vdom + diff 更快，但手动管理 DOM 的复杂度随项目规模指数增长——vdom 是"性能足够好 + 开发体验更好"的权衡。

#### 5. React Fiber

Fiber 是 React 16 引入的可中断异步渲染架构：
- 将渲染任务拆成小时间片（每个 ~5ms）
- 每个时间片结束检查是否有更高优先级任务（用户交互、动画）
- 有则中断当前渲染，执行高优任务，之后再恢复
- 实现 `requestIdleCallback` 类似的效果（但更可控）

#### 6. 为什么不能用 index 作为 key

```javascript
// 场景：列表 [A, B, C]，在头部插入 D
// index 作 key:  D(key=0) A(key=1) B(key=2) C(key=3)
//                  ↕          ↕         ↕         ↕
//           旧:   A(key=0)   B(key=1)  C(key=2)  (key=3 新增)
// → React 认为 A→D, B→A, C→B 都变了，全部重新渲染！
// 这还导致非受控组件的状态错乱（比如 input 的内容跟着 key 走）
```

### 五、性能优化高频题

#### 1. 首屏加载优化

| 策略 | 手段 |
|------|------|
| **减少资源体积** | 代码分割（Code Splitting）、Tree Shaking、Gzip/Brotli 压缩、图片 WebP |
| **减少网络请求** | HTTP/2 多路复用、雪碧图、关键 CSS 内联、字体子集化 |
| **加快资源加载** | CDN、预加载（`<link rel="preload">`）、DNS Prefetch |
| **优化渲染路径** | SSR/SSG、骨架屏、虚拟列表 |
| **缓存策略** | 强缓存 + 协商缓存、Service Worker 离线缓存 |

#### 2. 回流 & 重绘

- **回流（Reflow）**：修改几何属性（宽高/位置/display）→ 重新布局 → 开销大
- **重绘（Repaint）**：修改外观（颜色/背景/阴影）→ 不重新布局 → 开销较小
- **优化**：
  - 动画用 `transform` / `opacity`（只触发 Composite，不回流不重绘）
  - 批量 DOM 操作用 `documentFragment` 或 `display:none` 离线操作
  - 读写分离，避免强制同步布局

#### 3. 虚拟列表

10000 条数据 → 只渲染可视区域 + 缓冲区（约 20-30 个 DOM 节点），滚动时动态替换节点内容和位置。

```
┌──────────────────────┐
│  ┌────────────────┐  │ ← 占位容器（总高度 = itemHeight * totalCount）
│  │  item 7        │  │ ← 可视区起点 = scrollTop
│  │  item 8        │  │
│  │  item 9        │  │ ← 只渲染这 3 个（实际多一点做缓冲区）
│  │  item 10       │  │
│  └────────────────┘  │
└──────────────────────┘
```

### 六、工程化高频题

#### 1. Vite vs Webpack

| 维度 | Webpack | Vite |
|------|---------|------|
| 开发启动 | 全量打包 → 慢（大型项目几十秒） | ESM 按需编译 → 极快（秒启） |
| 热更新 | 随着模块增多变慢 | 精准更新，快 |
| 预构建 | 无 | esbuild 预构建 node_modules |
| 生产打包 | webpack + terser | Rollup + esbuild/terser |
| 插件生态 | 极其丰富（10年积累） | 快速追赶中 |
| 底层原理 | JS Bundle（把一切打包成 JS） | ESM（浏览器原生模块） |

**核心原理**：Vite 开发时直接把 `.vue`/`.ts` 文件在**浏览器请求时才编译**，由 esbuild 处理依赖预构建，不需要像 Webpack 那样把所有文件打包好再启动 dev server。

#### 2. Tree Shaking 原理

- 依赖 **ESM 的静态结构**（`import`/`export` 在编译期就能确定依赖关系）
- Webpack/Rollup 在打包时分析模块的 import/export，**标记未使用的导出为 dead code**
- Terser 等压缩工具再把这些 dead code 删除
- `package.json` 的 `"sideEffects": false` 告诉打包器"这个包没有副作用，放心 shake"

#### 3. Webpack Loader vs Plugin

| | Loader | Plugin |
|------|-------|--------|
| 作用 | **转换文件内容** | **扩展构建能力** |
| 执行时机 | 模块加载时 | 整个构建生命周期 |
| 示例 | `css-loader`、`babel-loader`、`vue-loader` | `HtmlWebpackPlugin`、`MiniCssExtractPlugin` |
| 本质 | 函数：`content → newContent` | 类：通过钩子（hooks）介入编译流程 |

### 七、TypeScript 高频题

#### 1. `any` vs `unknown`

```typescript
let a: any = 'hello'
a.toFixed()          // ❌ 运行时崩溃！any 绕过了类型检查

let b: unknown = 'hello'
b.toFixed()          // ❌ 编译报错！unknown 必须先做类型收窄
if (typeof b === 'number') {
  b.toFixed()        // ✅ 安全
}
```

**核心**：`any` 关闭类型检查，`unknown` 强制你检查后才使用——`unknown` 是"安全的 any"。

#### 2. 常用 Utility Types

```typescript
type Partial<T>   = { [P in keyof T]?: T[P] }          // 全部可选
type Required<T>  = { [P in keyof T]-?: T[P] }         // 全部必填
type Readonly<T>  = { readonly [P in keyof T]: T[P] }  // 全部只读
type Pick<T, K>   = { [P in K]: T[P] }                   // 选取部分属性
type Omit<T, K>   = { [P in Exclude<keyof T, K>]: T[P] } // 排除部分属性
type Record<K, V> = { [P in K]: V }                       // 构造对象类型
type Exclude<T, U> = T extends U ? never : T              // 从联合类型中排除
type Extract<T, U> = T extends U ? T : never               // 从联合类型中提取
```

#### 3. `interface` vs `type`

| | interface | type |
|------|---------|------|
| 声明合并 | ✅ 同名自动合并 | ❌ 不支持 |
| 继承方式 | `extends` | `&` 交叉类型 |
| 联合类型 | ❌ 不支持 | ✅ `type A = B \| C` |
| 元组/映射类型 | ❌ 不支持 | ✅ |
| 适用场景 | 对象结构（推荐） | 联合/交叉/工具类型 |
